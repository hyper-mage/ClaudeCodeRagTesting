# Module 7: Text-to-SQL + Web Search (Additional Tools)

## Complexity: ‚öÝÔ∏è Medium
9 tasks across 5 waves. Core challenge is safe SQL execution via a Postgres RPC, multi-tool routing in the chat router, and frontend changes for tool attribution. Each piece is straightforward individually but they wire together across the full stack.

## Context

Modules 1‚Äì6 built a full RAG pipeline with a single tool: `search_documents`. Module 7 adds two new tools, transforming the system from a single-tool agent into a **multi-tool agent**:

1. **`query_database`** ‚Äî Text-to-SQL over the user's own structured data (threads, messages, documents). The LLM generates SQL, executed safely via a read-only Postgres RPC that enforces RLS.
2. **`web_search`** ‚Äî Searches the web when uploaded documents can't answer the question. Provides attribution via source URLs.

**Key architectural decisions:**

- **Text-to-SQL safety:** A Postgres RPC (`execute_readonly_query`) wraps user-generated SQL in a read-only transaction, blocks non-SELECT statements, and enforces RLS. The schema is injected into the tool description so the LLM knows what columns exist.
- **Web search provider:** Tavily (purpose-built for RAG, returns clean snippets). Raw httpx like `embedding_service.py`. Tool only available when `WEB_SEARCH_API_KEY` is set.
- **Multi-tool routing:** The LLM handles routing natively via function calling ‚Äî no custom routing logic. The chat router's single `if` becomes a dispatch dict.
- **Frontend attribution:** Tool badges on assistant messages + markdown rendering for tables and clickable source links.

**Queryable schema (curated subset):**
- `threads`: id, title, created_at, updated_at
- `messages`: id, thread_id, role, content, created_at
- `documents`: id, filename, file_size, mime_type, status, chunk_count, metadata, created_at
- `document_chunks`: id, document_id, content, chunk_index, metadata

Excluded: `user_id` (implicit via RLS), `embedding` (binary noise), `storage_path`/`content_tsv` (internal).

### Task 2: Safe SQL Execution RPC

**Goal:** Postgres function that executes arbitrary SELECT queries safely with RLS enforcement.

**File:** `supabase/migrations/015_execute_readonly_query.sql` (new)

The function should:
1. Strip whitespace and validate query starts with `SELECT` or `WITH`
2. Block dangerous keywords (INSERT, UPDATE, DELETE, DROP, ALTER, CREATE, TRUNCATE, etc.)
3. Set RLS context (`request.jwt.claim.sub` + `SET LOCAL role = 'authenticated'`) so the query only sees the calling user's data
4. Execute with row limit via `LIMIT` wrapper
5. Return results as JSONB array
6. Use `SECURITY DEFINER` + immediate role switch for RLS enforcement

**Validation (run in Supabase SQL Editor):**
1. `SELECT execute_readonly_query('SELECT id, title FROM threads', 10, '<user_uuid>')` ‚Äî returns user's threads
2. `SELECT execute_readonly_query('DROP TABLE threads', 10, '<user_uuid>')` ‚Äî raises error
3. Query with user A's ID should NOT see user B's data

## Wave 3: Multi-Tool Chat Router

### Task 5: Tool Definitions + Chat Router Dispatch

**Goal:** Define two new tools, build a dispatch mechanism, implement dynamic tool availability. This is the core multi-tool agent teaching moment.

**File:** `backend/routers/chat.py` (modify)

**New tool definitions (module-level constants alongside `RETRIEVAL_TOOL`):**

`SQL_TOOL` ‚Äî `query_database` function. Description includes the schema string from `get_queryable_schema()`. Single required param: `sql` (string). Description tells LLM that RLS auto-filters to user's data ‚Äî don't add user_id filters.

`WEB_SEARCH_TOOL` ‚Äî `web_search` function. Single required param: `query` (string). Description says to use when docs don't have the answer or for recent/external info.

**Tool availability (in `event_generator()`):**
```python
tools = []
if doc_check.data:
    tools.append(RETRIEVAL_TOOL)
tools.append(SQL_TOOL)          # always available
if settings.web_search_enabled:
    tools.append(WEB_SEARCH_TOOL)
tools = tools if tools else None
```

**Dispatch refactor:** Extract an `execute_tool(fn_name, fn_args, user_id) -> str` helper that replaces the current inline `if fn_name == "search_documents"` block. The helper dispatches to the appropriate service and returns a JSON string result. The existing `search_documents` logic moves into this helper alongside the new `query_database` and `web_search` cases.

**Validation:**
1. Without web search key: LLM sees `search_documents` (if docs) + `query_database`
2. With web search key: all three tools available
3. Ask "how many documents have I uploaded?" ‚Äî LLM uses `query_database`
4. Ask about doc content ‚Äî LLM uses `search_documents`
5. Ask about current events ‚Äî LLM uses `web_search`

## Wave 4: SSE + Frontend Tool Indicators

### Task 6: Stream Tool Usage Events via SSE

**Goal:** Emit tool usage metadata to the frontend after each tool execution.

**File:** `backend/routers/chat.py` (modify ‚Äî within `event_generator`)

After each tool call executes, yield a new SSE event:
```python
yield {
    "event": "tool_event",
    "data": json.dumps({
        "tool_event": True,   # distinguishing field for frontend
        "tool": fn_name,
        "args_preview": <concise preview of args>,
    }),
}
```

The `tool_event: True` field is important because the current frontend SSE parser only looks at `data:` lines and identifies event types by checking JSON shape (not the SSE `event:` field). Adding this field lets the frontend detect tool events without changing the parsing approach.

**Validation:** Trigger a tool call ‚Äî `tool_event` appears in SSE stream with correct tool name
