# Module 8: Sub-Agents

## Complexity: ⚠️ Medium

5 tasks across 3 waves. One new service file, modifications to 4 existing files. The main challenge is wiring the sub-agent into the existing agentic loop cleanly and giving the frontend a differentiated display for sub-agent activity.

## Context

Modules 1-7 built a multi-tool RAG agent with `search_documents`, `query_database`, and `web_search`. These tools retrieve **chunks** or external data. But when a user asks "summarize my report.pdf" or "what are the key findings in document X?", chunk-level retrieval isn't enough — the LLM needs the **entire document** in context.

Module 8 adds an `analyze_document` tool that spawns an **isolated sub-agent**: a separate LLM call with the full document loaded into context and a specialized system prompt. The sub-agent's analysis is returned to the main agent as a tool result, which then crafts the final response.

**Key design decisions:**
- **Non-streaming sub-agent** — The sub-agent output is consumed by the main agent as a tool result, not shown directly to the user. Streaming it would add recursive SSE complexity for little benefit. Instead, we emit start/complete events so the frontend shows a spinner.
- **No sub-agent tools** — The sub-agent gets the full document text + a focused prompt. If cross-referencing is needed, the main agent can call `web_search` or `search_documents` separately after receiving the analysis.
- **LLM decides routing** — Just like existing tools, the LLM chooses `analyze_document` vs `search_documents` via function calling. The system prompt guides this choice.

### Task 2: Sub-Agent Service (`backend/services/subagent_service.py` — new)

**Reuse:** `get_supabase()` from `database.py`, `get_llm_client()` from `llm_service.py`, `get_settings()` from `config.py`

Three functions:

**`resolve_document(user_id, document_name) -> dict | None`**
- Query `documents` table with `user_id` filter + case-insensitive `ilike` on `filename`
- If multiple matches, return list of filenames in result for LLM to disambiguate
- Return document row dict or None

**`get_full_document_text(user_id, document_id) -> str`**
- Query `document_chunks` filtered by `document_id` and `user_id`, ordered by `chunk_index`
- Concatenate all chunk `content` fields with `\n\n`
- Return assembled text

**`run_document_analysis(user_id, document_name, analysis_query) -> dict`**
- Call `resolve_document` — if not found, return `{"error": "Document not found: ..."}`
- If multiple matches, return `{"error": "Multiple documents match: ...", "matches": [...]}`
- Call `get_full_document_text` — if text exceeds `subagent_max_context_chars`, truncate with warning
- Build messages: system prompt from settings + user message with full doc text + query
- Non-streaming `client.chat.completions.create()` via `get_llm_client()`, with `max_tokens` from settings
- Return `{"document": filename, "chunk_count": N, "analysis": response_text}`

```python
for tc in event["tool_calls"]:
    fn_name = tc["function"]["name"]
    fn_args = json.loads(tc["function"]["arguments"])

    # Emit sub-agent start event (spinner in frontend)
    if fn_name == "analyze_document":
        yield {
            "event": "tool_event",
            "data": json.dumps({
                "tool_event": True,
                "tool": "analyze_document",
                "subagent": True,
                "status": "running",
                "args_preview": fn_args.get("document_name", ""),
            }),
        }

    tool_result = execute_tool(fn_name, fn_args, user_id)

    current_messages.append({
        "role": "tool",
        "tool_call_id": tc["id"],
        "content": tool_result,
    })

    # Emit standard tool_event (completion)
    args_preview = fn_args.get("query", fn_args.get("sql", ""))[:100]
    tool_event_data = {
        "tool_event": True,
        "tool": fn_name,
        "args_preview": args_preview,
    }
    if fn_name == "analyze_document":
        tool_event_data["subagent"] = True
        tool_event_data["status"] = "complete"
    yield {
        "event": "tool_event",
        "data": json.dumps(tool_event_data),
    }
```

**4b. Update SSE handling** in `useChat.ts` (~line 84). When a sub-agent "complete" event arrives, update the existing "running" entry instead of adding a duplicate:

```typescript
if (parsed.tool_event === true) {
  setMessages(prev =>
    prev.map(m => {
      if (m.id !== assistantId) return m

      // Sub-agent complete: update existing running entry
      if (parsed.subagent && parsed.status === 'complete') {
        const updated = (m.toolsUsed || []).map(t =>
          t.tool === parsed.tool && t.subagent
            ? { ...t, status: 'complete' as const }
            : t
        )
        return { ...m, toolsUsed: updated }
      }

      // New tool event (including sub-agent start)
      return {
        ...m,
        toolsUsed: [
          ...(m.toolsUsed || []),
          {
            tool: parsed.tool,
            args_preview: parsed.args_preview,
            ...(parsed.subagent ? { subagent: true, status: parsed.status } : {}),
          },
        ],
      }
    })
  )
}
```
