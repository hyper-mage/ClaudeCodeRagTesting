# Module 6: Hybrid Search & Reranking

## Complexity: :warning: Medium
8 tasks across 4 waves. Core challenge is the SQL migration + Python RRF fusion logic; reranking is additive and optional.

## Context

Modules 1-4 built the full RAG pipeline with pure vector search via `match_document_chunks` RPC (cosine similarity on pgvector). Pure vector search has blind spots: it misses exact keyword matches (error codes, product names, acronyms) because embeddings capture semantic meaning, not lexical overlap.

Module 6 adds **Postgres full-text search** (tsvector/tsquery), combines both result sets using **Reciprocal Rank Fusion (RRF)**, and optionally **reranks** fused results for higher precision.

**Key decisions:**
- Keyword search via Postgres tsvector — no new infrastructure needed
- RRF fusion in Python — `score = sum(weight / (k + rank))` per document across lists
- Configurable reranking — LLM-based (reuses existing `get_llm_client()`) or dedicated rerank API; can be disabled
- `search_documents()` signature is **unchanged** — hybrid search is transparent to the chat router and LLM

> **Migration numbering note:** Current highest is `012`. If Module 5 adds migrations, adjust 013/014 accordingly.

### Task 2: Full-Text Search Migration

**Goal:** Add `content_tsv` tsvector column, backfill existing chunks, GIN index, and auto-populate trigger.

**Files:** `supabase/migrations/013_add_fulltext_search.sql` (new)

```sql
-- 1. Add tsvector column
ALTER TABLE document_chunks ADD COLUMN IF NOT EXISTS content_tsv tsvector;

-- 2. Backfill existing chunks
UPDATE document_chunks SET content_tsv = to_tsvector('english', content) WHERE content_tsv IS NULL;

-- 3. GIN index for fast full-text search
CREATE INDEX IF NOT EXISTS idx_document_chunks_content_tsv ON document_chunks USING GIN (content_tsv);

-- 4. Trigger to auto-populate on INSERT/UPDATE
CREATE OR REPLACE FUNCTION document_chunks_tsv_trigger() RETURNS trigger AS $$
BEGIN
  NEW.content_tsv := to_tsvector('english', NEW.content);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_document_chunks_tsv ON document_chunks;
CREATE TRIGGER trg_document_chunks_tsv
  BEFORE INSERT OR UPDATE OF content ON document_chunks
  FOR EACH ROW EXECUTE FUNCTION document_chunks_tsv_trigger();
```

- Uses `'english'` language config for stemming
- Trigger means **no changes needed to ingestion service** — tsvector is populated automatically
- Backfill handles all pre-existing chunks in one shot

**Validation:**
1. Run migration, then: `SELECT count(*) FROM document_chunks WHERE content_tsv IS NULL;` — should be 0
2. GIN index exists: check `pg_indexes`
3. Upload a new doc — new chunks have `content_tsv` populated automatically

## Wave 2: Keyword Search RPC + Rerank Service

### Task 3: Keyword Search RPC Function

**Goal:** Create a Postgres function for full-text keyword search, parallel to `match_document_chunks`.

**Files:** `supabase/migrations/014_keyword_search_function.sql` (new)

```sql
CREATE OR REPLACE FUNCTION keyword_search_chunks(
  search_query TEXT,
  match_count INT DEFAULT 20,
  filter_user_id UUID DEFAULT NULL,
  filter_metadata JSONB DEFAULT NULL
)
RETURNS TABLE (
  id UUID, document_id UUID, content TEXT, chunk_index INT, rank FLOAT
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT dc.id, dc.document_id, dc.content, dc.chunk_index,
    ts_rank_cd(dc.content_tsv, websearch_to_tsquery('english', search_query))::FLOAT AS rank
  FROM document_chunks dc
  WHERE dc.user_id = filter_user_id
    AND dc.content_tsv @@ websearch_to_tsquery('english', search_query)
    AND (filter_metadata IS NULL OR dc.metadata @> filter_metadata)
  ORDER BY rank DESC
  LIMIT match_count;
END;
$$;
```

- `websearch_to_tsquery` supports Google-style syntax (quotes, `-`, OR) — natural for LLM-generated queries
- `ts_rank_cd` (cover density) gives better ranking than basic `ts_rank`
- Returns `rank` (not `similarity`) to differentiate from vector results
- Same `filter_metadata` pattern as existing vector RPC

**Validation:**
1. Call: `SELECT * FROM keyword_search_chunks('some term', 5, '<user_id>');` — returns ranked results
2. Query with no matches — returns empty (no error)
3. With metadata filter — only matching chunks returned

## Wave 3: Hybrid Retrieval Service

### Task 5: Hybrid Retrieval with RRF Fusion

**Goal:** Rewrite `retrieval_service.py` to support all search modes with RRF fusion and optional reranking. **Crucially, the `search_documents()` signature stays the same.**

**Files:** `backend/services/retrieval_service.py` (rewrite)

**New internal functions:**
- `vector_search(user_id, query, top_k, metadata_filter)` — extracts current logic, calls `match_document_chunks` RPC
- `keyword_search(user_id, query, top_k, metadata_filter)` — calls new `keyword_search_chunks` RPC
- `reciprocal_rank_fusion(result_lists, weights, k=60)` — pure function. Deduplicates by chunk `id`, sums weighted `1/(k + rank + 1)` scores, returns sorted results

**Updated `search_documents()` pipeline:**
1. Determine `candidate_k = max(top_k * 4, 20)` — broader candidate pool for fusion/reranking
2. Based on `search_mode`: run vector only, keyword only, or both + RRF fusion
3. Pass through `rerank(query, results)` — no-ops if disabled
4. Trim to final `top_k`

**Key property:** If keyword search returns nothing (query terms not in any chunk), RRF gracefully degrades to vector-only results.

**Validation:**
1. `SEARCH_MODE=vector_only` — identical behavior to pre-Module 6 (regression test)
2. `SEARCH_MODE=keyword_only` — keyword matches ranked by text relevance
3. `SEARCH_MODE=hybrid` — fused results; exact-term queries surface correct chunks better than vector alone
4. `RERANK_ENABLED=true` — results reordered by relevance
5. Signature unchanged — chat router works without modification
