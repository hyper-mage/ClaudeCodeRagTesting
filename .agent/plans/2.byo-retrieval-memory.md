# Module 2: BYO Retrieval + Memory

## Context

Module 1 is complete — chat works via OpenAI's Responses API (managed threads/assistants). This is a black-box approach locked to OpenAI. Module 2 does two things:

1. **Provider transition** — Replace Responses API with standard Chat Completions API, supporting any OpenAI-compatible backend (OpenRouter default, plus Ollama, LM Studio, etc.)
2. **Ingestion pipeline** — File upload UI, chunking → embedding → pgvector storage, retrieval via tool calling, realtime status updates

**Key architectural shifts:**
- **Managed → stateless**: We load message history from our DB and send it with each request
- **Black-box RAG → BYO retrieval**: We control chunking, embedding, and vector search ourselves
- **Single provider → any provider**: OpenAI SDK with configurable `base_url`

**Decision: Option A (Replace)** — completely remove Responses API code, rebuild on Chat Completions.

## Complexity: 🔴 Complex

11 tasks organized into 6 execution waves. Tasks within the same wave can run in parallel. Consider breaking execution into sub-sessions at wave boundaries.

```python
def get_llm_client() -> OpenAI:
    """Create OpenAI-compatible client for chat completions."""

def get_embedding_client() -> OpenAI:
    """Create OpenAI client for embeddings (may use different provider than chat)."""

def stream_chat_completion(
    messages: list[dict],
    tools: list[dict] | None = None,
```

### Task 6: Embedding Service

**Goal:** Create a service for generating embeddings via the configurable embedding provider.

**Depends on:** Task 2 (needs `get_embedding_client()` from `llm_service.py`)

**File:** `backend/services/embedding_service.py` *(new)*

```python
def get_embeddings(texts: list[str]) -> list[list[float]]:
    """Generate embeddings for a batch of texts using configured embedding provider.
    Works with OpenAI, Ollama, or any OpenAI-compatible embedding endpoint."""
    client = get_embedding_client()  # from llm_service
    settings = get_settings()
    response = client.embeddings.create(
        model=settings.embedding_model,
        input=texts,
    )
    return [item.embedding for item in response.data]
```

**Note:** Omit `dimensions` param when calling Ollama — not all providers support it. Only pass it for OpenAI. Could add a flag like `embedding_supports_dimensions: bool = True` to config, or detect from `base_url`.

**Validation:**
1. `get_embeddings(["hello world"])` returns a list of floats matching configured dimensions
2. LangSmith traces the embedding call

## Wave 4: Ingestion Backend

### Task 7: Ingestion Backend — Upload + Processing Pipeline

**Goal:** Backend endpoints for file upload and document processing (chunk → embed → store).

**Depends on:** Task 4b (tables), Task 5 (storage bucket), Task 6 (embedding service)

**Files:**
- `backend/routers/documents.py` *(new)* — document CRUD + upload endpoint
- `backend/services/ingestion_service.py` *(new)* — chunking + processing logic
- `backend/main.py` — register new router

**Endpoints:**
- `POST /api/documents/upload` — upload file, create document record, trigger processing
- `GET /api/documents` — list user's documents with status
- `GET /api/documents/{id}` — single document details
- `DELETE /api/documents/{id}` — delete document + chunks + storage file

**Upload flow (synchronous, designed for easy async migration):**
1. Accept file via `UploadFile` (FastAPI)
2. Upload to Supabase Storage at `{user_id}/{doc_id}/{filename}`
3. Create `documents` row (status: `pending`)
4. Call `process_document(doc_id, user_id)` — isolated function that can later be called via background task:
   - Update status to `processing`
   - Download file content from storage
   - Chunk text with configurable `chunk_size` and `chunk_overlap`
   - Generate embeddings for all chunks via `get_embeddings()` (batched)
   - Insert chunks into `document_chunks` table
   - Update document status to `completed` + set `chunk_count`
   - On error: set status to `failed`, populate `error_message`
5. Return document record

**Async-ready design:** The `process_document()` function is self-contained — it takes a `doc_id` and `user_id`, fetches everything it needs from DB/storage, and updates status via DB writes. To make it async later, just call it from `asyncio.create_task()` or a task queue instead of inline.

**Chunking strategy (Module 2):**
- Recursive character splitting: split on `\n\n`, then `\n`, then `. `, then characters
- Configurable `chunk_size` (default 1000) and `chunk_overlap` (default 200)

**File format support (Module 2):**
- Plain text (`.txt`) and Markdown (`.md`) only
- Module 5 adds PDF/DOCX/HTML via docling

**Validation:**
1. Upload a `.txt` file → document created, chunks stored with embeddings
2. `GET /api/documents` → shows document with `status: completed`
3. Check `document_chunks` table → chunks with embeddings present
4. Delete document → chunks cascade-deleted, storage file removed
5. Upload unsupported file type → status: `failed`, error_message populated

### Task 8: Retrieval Tool + Tool Calling in Chat

**Goal:** Add a `search_documents` tool to chat so the LLM can retrieve relevant chunks.

**Depends on:** Task 3 (chat router), Task 7 (needs ingested documents to search)

**Files:**
- `backend/services/retrieval_service.py` *(new)* — vector search function
- `backend/services/llm_service.py` — update `stream_chat_completion` for tool call handling
- `backend/routers/chat.py` — wire up tool execution loop
- `supabase/migrations/008_match_chunks_function.sql` *(new)* — vector search RPC

**Supabase RPC function:**
```sql
CREATE OR REPLACE FUNCTION match_document_chunks(
  query_embedding VECTOR(1536),
  match_count INT DEFAULT 5,
  filter_user_id UUID DEFAULT NULL
)
RETURNS TABLE (
  id UUID, document_id UUID, content TEXT, chunk_index INT, similarity FLOAT
)
LANGUAGE plpgsql AS $$
BEGIN
  RETURN QUERY
  SELECT dc.id, dc.document_id, dc.content, dc.chunk_index,
    1 - (dc.embedding <=> query_embedding) AS similarity
  FROM document_chunks dc
  WHERE dc.user_id = filter_user_id
  ORDER BY dc.embedding <=> query_embedding
  LIMIT match_count;
END;
$$;
```

**Retrieval function:**
```python
def search_documents(user_id: str, query: str, top_k: int = 5) -> list[dict]:
    query_embedding = get_embeddings([query])[0]
    results = db.rpc("match_document_chunks", {
        "query_embedding": query_embedding,
        "match_count": top_k,
        "filter_user_id": user_id,
    }).execute()
    return results.data
```

**Tool definition for Chat Completions:**
```python
RETRIEVAL_TOOL = {
    "type": "function",
    "function": {
        "name": "search_documents",
        "description": "Search the user's uploaded documents for relevant information",
        "parameters": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "Search query"}
            },
            "required": ["query"],
        },
    },
}
```

**Chat flow with tool calling:**
1. Send messages + `[RETRIEVAL_TOOL]` to Chat Completions (streaming)
2. If model returns a tool call → execute `search_documents(user_id, query)` → append tool result to messages → call Chat Completions again (stream final answer)
3. If model returns text → stream as before
4. SSE events stay the same — tool execution is invisible to the frontend

**Validation:**
1. Upload a document about a specific topic
2. Ask a question about that topic → LLM calls `search_documents` → answers with context from the document
3. Ask an unrelated question → LLM answers directly without tool call
4. LangSmith shows the tool call chain (user msg → tool call → tool result → assistant response)
