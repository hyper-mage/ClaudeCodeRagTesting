# Module 1: App Shell + Observability

## Context

This is the first module of the Agentic RAG Masterclass. We're building the foundational app shell — auth, chat UI, OpenAI Responses API integration (managed threads + file_search), and LangSmith observability. No application code exists yet; this is greenfield.

The Responses API is a managed solution where OpenAI handles threads, memory, and retrieval. Module 2 will replace it with the standard Chat Completions API for provider flexibility. The architecture here is designed so auth, UI, and database layers survive that transition untouched.

## Complexity: ⚠️ Medium

6 sequential tasks. Each is single-pass executable, but they must be done in order.

## Task 0: External Service Setup

**Goal:** Create Supabase project and obtain all API keys.

### Supabase Project
1. Go to [supabase.com](https://supabase.com), create a new project
2. Note down from Project Settings:
   - **Project URL** (`SUPABASE_URL`)
   - **Anon/public key** (`SUPABASE_ANON_KEY`)
   - **Service role key** (`SUPABASE_SERVICE_ROLE_KEY`)
   - **JWT Secret** (Settings > API > JWT Settings) (`SUPABASE_JWT_SECRET`)
3. Enable Email auth provider (Authentication > Providers — enabled by default)

### OpenAI API Key
1. Go to [platform.openai.com](https://platform.openai.com), create an API key
2. Ensure the account has access to the Responses API (requires a funded account)
3. Note down as `OPENAI_API_KEY`

### LangSmith API Key
1. Go to [smith.langchain.com](https://smith.langchain.com), create an account
2. Create a new API key (Settings > API Keys)
3. Create a project called `rag-masterclass`
4. Note down as `LANGSMITH_API_KEY`

**Validation:** All 6 env vars collected and ready to paste into `.env`.

## Task 2: Supabase Database Schema & RLS

**Goal:** Create tables for threads and messages with Row-Level Security.

### Schema

**`threads` table:**
| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK (gen_random_uuid()) | |
| user_id | uuid NOT NULL (refs auth.users) | |
| title | text | Auto-generated from first message |
| openai_thread_id | text | Module 1 only — dropped in Module 2 |
| created_at | timestamptz (now()) | |
| updated_at | timestamptz (now()) | |

**`messages` table:**
| Column | Type | Notes |
|--------|------|-------|
| id | uuid PK (gen_random_uuid()) | |
| thread_id | uuid NOT NULL (refs threads ON DELETE CASCADE) | |
| user_id | uuid NOT NULL (refs auth.users) | |
| role | text NOT NULL CHECK ('user', 'assistant') | |
| content | text NOT NULL | |
| openai_message_id | text | Module 1 only |
| created_at | timestamptz (now()) | |

### RLS Policies (both tables)
- SELECT: `auth.uid() = user_id`
- INSERT: `auth.uid() = user_id`
- UPDATE: `auth.uid() = user_id` (threads only, for title)
- DELETE: `auth.uid() = user_id`

### Files
```

  001_create_threads.sql
  002_create_messages.sql
```

### Validation
1. Run SQL in Supabase SQL Editor — tables visible in Table Editor
2. RLS enabled on both tables (check dashboard)

## Task 3: Backend — Auth, Config & API Structure

**Goal:** FastAPI app with JWT verification, thread CRUD, and structured routing.

### Files
```
/backend/
  main.py                      # App, CORS, route registration
  config.py                    # Pydantic BaseSettings
  auth.py                      # JWT verification dependency
  database.py                  # Supabase client init (service role)
  routers/
    __init__.py
    threads.py                 # Thread CRUD
    chat.py                    # Chat endpoint (placeholder, built in Task 5)
  models/
    __init__.py
    schemas.py                 # Pydantic request/response models
```

### Auth Flow
1. Frontend authenticates via Supabase JS SDK (client-side)
2. Frontend sends JWT in `Authorization: Bearer <token>` on every request
3. Backend `auth.py` decodes JWT using `SUPABASE_JWT_SECRET`, extracts `user_id` from `sub` claim
4. All endpoints receive `user_id` via FastAPI dependency injection

### Thread Endpoints
- `GET /api/threads` — list user's threads (ordered by updated_at desc)
- `POST /api/threads` — create thread (+ OpenAI thread via Responses API)
- `GET /api/threads/{thread_id}` — thread with messages
- `DELETE /api/threads/{thread_id}` — delete thread (cascade messages)

Backend uses Supabase client with service role key, filtering by user_id explicitly (belt-and-suspenders with RLS).

### Validation
1. `GET /api/threads` with no auth → 401
2. With valid JWT → empty array
3. `POST /api/threads` → creates thread, `GET` returns it
4. Second user cannot see first user's threads

## Task 5: OpenAI Responses API + SSE Streaming

**Goal:** Chat endpoint that sends messages to OpenAI Responses API, streams response via SSE, stores messages in DB.

### Files
```
/backend/
  services/
    __init__.py
    openai_service.py          # OpenAI Responses API wrapper (streaming)
  routers/
    chat.py                    # POST /api/threads/{thread_id}/messages (SSE)

/frontend/src/
  hooks/
    useChat.ts                 # Message state, SSE connection, send logic
  components/
    ChatContainer.tsx          # Updated with useChat hook
    MessageBubble.tsx          # Streaming display (incremental text)
```

### Backend Flow (`POST /api/threads/{thread_id}/messages`)
1. Receive `{ content: string }`
2. Verify thread belongs to user
3. Store user message in `messages` table
4. If no `openai_thread_id`, create via `client.beta.threads.create()`
5. Add message to OpenAI thread
6. Create run with `stream=True`
7. Return `EventSourceResponse` (SSE)
8. Stream `content_delta` events with text chunks
9. On completion, send `done` event with full message
10. Store assistant message in DB

### SSE Event Format
```
event: content_delta
data: {"text": "Here is"}

event: content_delta
data: {"text": " the answer"}

event: done
data: {"message_id": "uuid", "content": "Here is the answer"}
```

### Frontend Flow (`useChat.ts`)
- POST with `fetch` + `ReadableStream` reader (not `EventSource`, which is GET-only)
- Parse SSE frames, append deltas to in-progress message
- On `done`, finalize message in state

### Key Decisions
- Auto-generate thread title after first assistant response (first ~50 chars of user message, or LLM summary)
- Messages stored locally AND in OpenAI — local copy drives UI, OpenAI copy drives LLM context
- `openai_service.py` is the ONLY file that imports OpenAI — clean replacement target for Module 2

### Validation
1. Send message → streaming response appears token-by-token
2. DB has both user and assistant messages with correct thread_id/user_id
3. Page refresh → conversation history loads from DB
4. New thread → `openai_thread_id` populated in threads table
5. Network tab shows SSE stream with `content_delta` events

## Task 6: LangSmith Observability

**Goal:** Trace all OpenAI calls in LangSmith from day one.

### Files
```
/backend/
  services/
    tracing.py                 # LangSmith setup
    openai_service.py          # Updated to use traced client
```

### Approach
- Use `langsmith.wrappers.wrap_openai()` to patch the OpenAI client — automatically traces all SDK calls
- Use `@traceable` decorator on chat handler for a parent span grouping all OpenAI calls per request
- Env vars: `LANGCHAIN_TRACING_V2=true`, `LANGCHAIN_API_KEY`, `LANGCHAIN_PROJECT`
- Opt-in: if `LANGSMITH_API_KEY` is empty, tracing is disabled, app works normally

### Validation
1. Set LangSmith env vars, send a chat message
2. Open LangSmith dashboard → trace appears with full request/response
3. Parent trace shows chat handler, child traces show OpenAI calls
4. Remove API key → app still works, no errors

## Module 2 Readiness

These elements are designed to be swappable:
- `openai_thread_id` / `openai_message_id` columns → drop in Module 2

- Messages stored locally → no data migration needed
- `useChat.ts` SSE parsing → reusable, only backend endpoint changes
- Auth, UI, thread CRUD → completely independent of LLM layer
