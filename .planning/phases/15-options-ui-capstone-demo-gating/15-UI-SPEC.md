# Phase 15 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase scope (from 15-CONTEXT.md):** the v1.2 capstone. Five surfaces, all extending live components:

1. **Sectioned + searchable picker upgrade** (`ModelSelector.tsx`) — Favorites → Popular → All sections (D-06), a fuzzy search input pinned atop the dropdown (D-08, closes audit W-1 / MODEL-01), a `Popular` badge chip on rows (D-07, closes audit B-1 / MODEL-03), and a favorite-star affordance per row persisted to `user_preferences.favorite_models` (D-05, MODEL-08).
2. **Key-gate confirm modal** (KEY-05, D-01/D-03/D-04) — keyless selection on EITHER surface (thread-header `ModelSelector`, settings `DefaultModelSelector`) routes through one shared gate: paid pick (demo ON) or any pick (demo OFF) opens a confirm dialog whose [Connect] launches the existing PKCE flow.
3. **Post-OAuth auto-apply** (D-02) — the pending selection rides `sessionStorage` beside the PKCE verifier; `OAuthCallbackPage` applies it after a successful exchange (thread PATCH / prefs PUT) and fires a confirmation toast.
4. **Non-dismissible demo banner** (DEMO-02, D-10) — slim amber strip at the top of the chat pane with the LOCKED Phase-11 copy.
5. **[Use demo] recovery action** (D-11) — wire the dead `demoEligible`/`onUseDemo` props on `ErrorMessageBubble` (403 path) to real demo-flag state; clicking retries the turn in demo mode.

**Explicitly NOT this phase** (do not design): FU-A/FU-B/FU-D Phase-14 follow-ups (balance semantics, post-turn balance refresh, **light-mode bubble polish — do not expand FU-D scope**), W-3 notice dedup, aux/utility-model override UI, MODEL-F1 "New" badge, MODEL-F2 heavy keyboard nav / drag-reordering, admin UI for the demo flag (env-driven only), SEC-01 human gates and prod migration catch-up (deploy-track, not build scope).

## Spacing Scale

Declared values (multiples of 4). Inherits the Phase-14 scale; Phase-15 usage noted.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (`gap-1`), chip horizontal padding (`px-1`), row sub-line gap (`gap-0.5` existing) |
| sm | 8px | Row/button gaps (`gap-2`), banner vertical padding (`py-2`), search-icon inset |
| md | 12px | Dropdown row + search-input horizontal padding (`px-3`), modal button padding (`px-4 py-2` existing) |
| lg | 16px | Banner horizontal padding (`px-4`), chat-pane padding (existing `p-4`) |
| xl | 24px | Modal inner padding (`p-6`, existing ConfirmDialog), modal button-row top margin (`mt-6`) |
| 2xl | 32px | Reserved (not needed this phase) |
| 3xl | 64px | Reserved (not needed this phase) |

**Touch targets — 44px minimum everywhere interactive:**
- Picker trigger: existing `min-h-11` (unchanged).
- Dropdown option rows: existing `min-h-11` (unchanged).
- **Search input row: `h-11`** (new — do not ship a shorter input).
- **Favorite-star button: 44px hit area** — `absolute right-0 inset-y-0 w-11 flex items-center justify-center` inside the ≥44px row; the visible `Star` icon is 16px. Row content gets `pr-12` clearance so text never sits under the star.
- Gate-modal buttons: existing ConfirmDialog `px-4 py-2` + `text-sm` (~36px) is the established modal-button size — acceptable as-is (matches disconnect dialog); do not shrink further.

Exceptions:
- The **demo banner** is non-interactive (non-dismissible by definition) — not a touch target.
- **Chips** (`Free`, `Popular`) and **section headers** are non-interactive caption elements.
- Legacy sub-4 paddings already in the codebase (`py-0.5` chips, `py-2.5` recovery buttons) remain valid existing patterns.

## Interaction Contract — picker a11y (LOCKED, supersedes the listbox-focus model)

Adding the search input changes the focus model from "focus the UL" to the **combobox-with-list-popup pattern** (what cmdk/shadcn implement):

- Trigger button: unchanged — `aria-haspopup="listbox"`, `aria-expanded`, `aria-controls={listboxId}`; Enter/Space/ArrowDown opens.
- On open: focus moves to the **search input** (not the UL). Input carries `aria-autocomplete="list"`, `aria-controls={listboxId}`, `aria-activedescendant={activeOptionId}`.
- Options get stable ids (`${listboxId}-opt-${key}`); list keeps `role="listbox"`, rows `role="option"` + `aria-selected`; section headers `role="presentation"`.
- Keys on the input: **ArrowDown/ArrowUp** move the active row (skipping headers, wrapping NOT required); **Enter** selects the active row; **Shift+Enter** toggles favorite on the active row; **Esc** closes and returns focus to the trigger; **Tab** stays trapped (preventDefault, existing behavior); Home/End operate on the text caret (native input behavior — no list jump). All printable keys type into the filter.
- Mouse: hover sets active; row click selects; star click toggles without selecting or closing; outside-click closes without returning focus.
- The star button is `tabIndex={-1}` (mouse/touch + Shift+Enter reach it; it is never a tab stop inside the trap) with a per-row `aria-label`.
- Existing `ModelSelector.test.tsx` assertions against the UL-focus model will need updating to the input-focus model — an expected, contract-sanctioned change.

## Interaction Contract — keyless gating (D-03 interpretation, flagged for checker)

CONTEXT D-03 contains a tension: "picker filters via `?free_only=true`" vs. "Paid rows → connect modal (D-01)" in the same demo-ON bullet — a display filter would leave no paid rows to gate, and since prod ships demo-ON (D-09), hiding paid rows would make KEY-05's picker trigger unreachable in the shipped config. **This contract resolves it as: the full catalog is always rendered; row-level `is_free` (server-computed, rendered verbatim — never recomputed client-side) drives the gate branch.** Free rows apply immediately for keyless demo-ON users; paid rows open the gate modal (KEY-05's primary path); demo-OFF gates every row. `GET /api/models?free_only=true` remains available to the planner as an implementation aid (e.g. server-verified free set for tests or the backend guard's sibling), but it is NOT a display contract. The server-side free-guard in the demo resolution branch (D-03, non-negotiable) is unaffected. If the checker or user overrides toward a free-only display, KEY-05 needs an alternate keyless→OAuth picker affordance — call it out rather than silently dropping the requirement.
