## Summary

This phase delivers a one-shot, out-of-band Python script in `backend/scripts/` that deliberately spends real money on the **prod owner OpenRouter account** using the paid model `openai/gpt-4o-mini` until the account's pre-configured **$0.10 cost guardrail** trips, then records the empirical outcome (which HTTP status the block returns, whether an email is sent) as the SEC-03 evidence artifact. The script is pure ops tooling — it does **not** touch the FastAPI request path; it authenticates with the owner key directly and calls OpenRouter's chat-completions endpoint, mirroring the existing `verify_langsmith_routing.py` / `seed_model_cache.py` standalone-script bootstrap (sys.path insert → `from config import get_settings`), targeting prod via `ENV_FILE=.env.prod`.

The single most important finding is a **documented contradiction in OpenRouter's own docs** about what a budget/spend guardrail returns when it trips: the Guardrails *overview* page says **403 Forbidden** ("budget limits and allowlist restrictions also produce 403 responses"), while the Guardrails *announcement blog* says **402** ("requests that exceed the limit for the time period will fail with a `402` response"). The general error reference reserves 402 for account-level "insufficient credits" and 403 for "guardrail block." Because the docs disagree, the script must **not** hard-code a single expected status — it must treat **402 OR 403 as a confirmed block** (SEC-03 PASS for the block condition), treat **429 as rate-limiting (back off, NOT a guardrail trip)**, and record the *actual* observed status + scrubbed body as the empirical artifact. This ambiguity is exactly why an empirical trip-test exists.

Two secondary findings de-risk the build: (1) **`usage.cost` is now returned on every completion automatically** — the old `usage:{include:true}` opt-in is deprecated and has no effect — so per-call USD cost is free to read and is the cleanest running-total source; and (2) the **app-level kill switch (D-05a) is already proven** by existing unit tests (`test_no_key_flag_off_refuses`, `test_fail_closed_no_or_fallback`) against `_resolve_key_and_model`, so no live spend is needed for that half of SEC-03.

**Primary recommendation:** Build a standalone `backend/scripts/burn_owner_guardrail.py` (run with `ENV_FILE=.env.prod` + an explicit `--i-understand-this-spends-real-money` confirm flag). Mint **serial** `openai/gpt-4o-mini` completions in small batches, summing per-response `usage.cost` as the primary running total and reconciling against `GET /api/v1/key`.`data.usage` between batches. Stop on the FIRST `openai.APIStatusError` with status 402/403 (block confirmed → record code+scrubbed body), back off on 429, and **hard-abort at $0.25 cumulative** (→ SEC-03 BLOCKER). Scrub all output with the existing `scrub_secrets`. Prove the app-level kill switch via the existing resolution tests; unit-test the script's spend-math / abort / block-classification as pure functions (zero spend). The live trip itself is a manual operator run, documented as such.

### Pattern 3: Spend poll between batches (mirror `keys.py` balance proxy)
**What:** `GET https://openrouter.ai/api/v1/key` with `Authorization: Bearer <owner key>`; read `data.usage` (cumulative USD on the key) and `data.limit_remaining`.
**When to use:** Once per batch as a reconciliation cross-check against the per-call `usage.cost` sum.
**Example:**
```python
# Source: backend/routers/keys.py:135-151 (VERIFIED, mirrored — note nested {"data": {...}})
import httpx
r = httpx.get("https://openrouter.ai/api/v1/key",
              headers={"Authorization": f"Bearer {owner_key}"}, timeout=15)
r.raise_for_status()
data = r.json().get("data") or {}
server_usage = data.get("usage")            # total OpenRouter credit usage (USD) for the key
limit_remaining = data.get("limit_remaining")  # may be None for uncapped keys; informational
```
> NOTE: `limit`/`limit_remaining` on `/api/v1/key` reflect the **key's own** spending limit, which may NOT equal the **org/member Guardrail budget** (guardrail budgets are "enforced per-user and per-key, not shared"). Treat `data.usage` (cumulative spend) as the reliable cross-check; treat `limit_remaining` as informational only. [CITED: openrouter.ai/docs/guides/features/guardrails/overview]

### Pattern 4: Block classification (the heart of D-04)
**What:** Catch `openai.APIStatusError`; branch on `.status_code`.
**When to use:** Around every `create()` call.
**Example:**
```python
import openai
try:
    resp = client.chat.completions.create(...)
except openai.APIStatusError as e:
    code = e.status_code
    body = scrub_secrets(getattr(e, "response", None).text if e.response else str(e))
    if code in (402, 403):
        # GUARDRAIL BLOCK CONFIRMED — record code+body, this is the SEC-03 trip signal
        return record_block(code, body, running_total)
    elif code == 429:
        # rate-limited — back off and retry; this is NOT the cost guardrail
        backoff(); continue
    else:
        raise   # 400/401/5xx etc. — unexpected, surface it (scrubbed)
```

### Anti-Patterns to Avoid
- **Calling `resolved_llm_api_key()` as a method.** It is a `@property` — `settings.resolved_llm_api_key()` raises `TypeError: 'str' object is not callable`. CONTEXT.md wrote it with parens; that is wrong. Access it bare. [VERIFIED: backend/config.py:182-185, used bare in chat.py:200 & llm_service.py:23]
- **Hard-coding the expected trip status (402 only, or 403 only).** OpenRouter's own docs disagree (see Open Questions). Accept BOTH 402 and 403 as "blocked."
- **Treating 429 as the guardrail trip.** 429 is rate-limiting; misreading it as the cost cap would produce a false SEC-03 PASS. Distinguish explicitly.
- **Big parallel fan-out.** D-06 forbids it — overshoot past $0.10 would be unbounded. Issue serially in small batches with a between-batch spend re-check.
- **Setting `os.environ["ENV_FILE"]` *after* `import config`.** `load_dotenv` runs at config import; a late set targets the wrong env. Set it in the shell or before the import.
- **Defaulting the script to prod silently.** Real money. Require an explicit confirm flag AND assert the loaded env is `.env.prod` before spending.
- **`trace=True` / LangSmith-wrapped client.** Burn-test traffic would pollute the prod LangSmith project. Use `trace=False` or a raw client.

## Common Pitfalls

### Pitfall 1: Expecting one specific HTTP status for the trip
**What goes wrong:** Script asserts `== 402` (or `== 403`) and the guardrail returns the other → false BLOCKER, or worse, the script never recognizes the block and keeps spending to $0.25.
**Why it happens:** OpenRouter's overview doc says 403 for budget limits; its announcement blog says 402; the error reference splits 402 (account credits) vs 403 (guardrail block). Genuinely ambiguous.
**How to avoid:** Treat 402 **and** 403 as "blocked." Record the actual code. This *is* the deliverable.
**Warning signs:** A test that pins a single status; spend climbing past $0.12 without a recognized block.

### Pitfall 2: Per-call cost too small → thousands of calls to reach $0.10
**What goes wrong:** `gpt-4o-mini` is cheap ($0.15/M in, $0.60/M out). A tiny prompt with a short answer costs ~$0.0001 → ~1000 calls to reach $0.10. CONTEXT's "~$0.005/call → 20-25 calls" assumes substantial tokens.
**Why it happens:** Default `max_tokens` + a one-line prompt produce near-zero cost per call.
**How to avoid:** Use a sizeable prompt and a high `max_tokens` (e.g., "write a long detailed essay…", `max_tokens` ~4000–8000), then make batch sizing **adaptive** from the observed `usage.cost`. Add a **max-call-count ceiling** (e.g., 300) as a third safety bound beside the $0.25 abort.
**Cost math (gpt-4o-mini @ $0.15/M in, $0.60/M out):** ~$0.0025/call at 4k output → ~40 calls; ~$0.0098/call at full 16,384 output → ~10 calls. Real output varies → adapt from `usage.cost`. [VERIFIED: openrouter.ai/openai/gpt-4o-mini pricing]

### Pitfall 3: Overshoot past $0.10 before the block lands
**What goes wrong:** With parallel/large batches, many calls succeed before the guardrail rejects, overshooting the cap.
**Why it happens:** The guardrail rejects the *next* request once usage reaches the limit; in-flight/concurrent calls already past that point still get charged.
**How to avoid:** Serial issuance + re-check between small batches. Worst-case overshoot ≈ (batch_size × max per-call cost). Shrink batch size as the total approaches $0.10. The blocked request itself is rejected pre-inference → $0, so natural overshoot is ~one successful call.
**Warning signs:** Final spend ≫ $0.11.

### Pitfall 4: ENV_FILE set too late → burns the wrong account
**What goes wrong:** `os.environ["ENV_FILE"]="..env.prod"` after `import config` has no effect (dotenv already loaded `.env` = dev). The script then either spends on dev or fails to find the prod key.
**Why it happens:** `config.py` calls `load_dotenv(...)` at module import, keyed on `ENV_FILE`. [VERIFIED: backend/config.py:7-10]
**How to avoid:** Set `ENV_FILE=.env.prod` in the shell before launch (recommended), and add a runtime PROD-assert in the script (e.g., refuse to run unless a prod-only marker resolves), plus the explicit confirm flag.
**Warning signs:** Spend not appearing on the prod account; `resolved_llm_api_key` empty.

### Pitfall 5: Leaking the owner key into output / traces
**What goes wrong:** Printing the client config, an unscrubbed error body, or a LangSmith-wrapped trace ships `sk-or-…` across the trust boundary (SEC-01).
**Why it happens:** Error bodies and debug prints are easy to forget; `get_llm_client(trace=True)` wraps with LangSmith.
**How to avoid:** Pipe every printed/logged string through `scrub_secrets`; build the client with `trace=False` (or raw `OpenAI(...)`); never print `settings.resolved_llm_api_key`.
**Warning signs:** `sk-or-` substring anywhere in stdout/stderr/logs.

### Pitfall 6: Account has insufficient real credits to reach $0.25
**What goes wrong:** If the prod owner account balance is below the spend needed, you may hit a genuine **402 "insufficient credits"** (account empty) instead of the **guardrail** trip — a different finding.
**Why it happens:** Guardrail budget ($0.10) is separate from account balance; if balance < budget the account-empty 402 fires first.
**How to avoid:** Confirm the owner account has at least ~$0.30 of spendable credit before running (D-03 says guardrail is live; also verify balance). Record whether the block was the guardrail vs account-empty (the body/message distinguishes them).
**Warning signs:** A 402 with an "insufficient credits / add credits" message rather than a budget/guardrail message at well below the configured cap.

## Open Questions

1. **Does the $0.10 budget guardrail trip with HTTP 402 or 403?**
   - What we know: OpenRouter's Guardrails *overview* doc says **403** ("budget limits and allowlist restrictions also produce 403 responses"); the Guardrails *announcement blog* says **402** ("requests that exceed the limit … fail with a `402` response"); the error reference reserves 402 for account-credit exhaustion and 403 for "guardrail block." The two OpenRouter sources directly contradict each other.
   - What's unclear: which one the live prod account actually returns.
   - Recommendation: **This is the phase's empirical deliverable.** Accept 402 OR 403 as "blocked"; record the actual observed code + scrubbed body. Do not let the planner pin a single code in a verification step.

2. **Does any email/notification fire on the trip?**
   - What we know: No mention of email in the errors doc, the guardrails overview, or the announcement blog; prior intel says Guardrails replaced the email-alert toggle.
   - What's unclear: whether a residual notification exists.
   - Recommendation: D-04(3) — record "no email" as the EXPECTED, ACCEPTED finding. The operator manually checks the OpenRouter account inbox/notifications after the run; the script cannot observe email and should say so.

3. **Does the org Guardrail budget surface in `/api/v1/key`.`limit_remaining`?**
   - What we know: `limit`/`limit_remaining` are the *key's* spending limit; guardrail budgets are "enforced per-user and per-key, not shared" — they may or may not map onto these fields.
   - Recommendation: Rely on `data.usage` (cumulative spend) for reconciliation; treat `limit_remaining` as informational. The definitive trip signal is the 402/403 on a completion, not a poll.
