<objective>
Wire auto-create-on-send into `ChatPage.handleSend` so a message sent while `activeThreadId === null` silently creates a thread, then sends into it — making the first message "just work" with a visible optimistic bubble (D-01), for both cold start and post-delete (D-04), backend-titled (D-03).

Purpose: This is the behavior fix at the page seam. Plan 01 made `sendMessage` accept an explicit `threadId`; this plan supplies that id from a freshly-created thread BEFORE the send, defeating the React stale closure (RESEARCH § Pitfall 1) that is the root cause of the dropped first message. It reuses the existing `handleNewThread` create pattern and the existing post-send `loadThreads()` title refresh — almost nothing here is net-new behavior; it unblocks existing machinery.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser SPA → `POST /api/threads` | Authenticated create; the new thread id is issued by the server for the authenticated user (RLS-scoped `user_id`). |
| browser SPA → `POST /api/threads/{id}/messages` (SSE) | The id used is the server-issued one from the create response — never client-minted, never user-supplied. |
| create-failure path → toast/Sentry | Error feedback surface; must not leak server error bodies into copy (reuse existing generic strings). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-06 | Elevation of Privilege (IDOR) | handleSend auto-create → sendMessage(threadId) | mitigate | targetId is exclusively the `thread.id` returned by the server's own `POST /api/threads` for the authenticated user; the messages endpoint re-checks `user_id` + RLS. No client-minted or user-supplied id is ever passed to sendMessage. |
| T-999.1-07 | Repudiation / Availability (silent failure) | create-failure path | mitigate | A failed `POST /api/threads` triggers Sentry.captureException + an error toast and aborts the send — the exact silent-no-op class this phase fixes is not reintroduced (RESEARCH § Pitfall 5). |
| T-999.1-08 | Information Disclosure | create-failure toast copy | mitigate | Reuse the existing generic error string; do NOT interpolate the caught error / HTTP status / response body into the toast (mirrors the locked Phase-10 generic-failure rule). |
</threat_model>
