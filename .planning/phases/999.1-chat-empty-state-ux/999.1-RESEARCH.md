## Summary

This is a **frontend-only React 19 UX fix**. The root cause is precise and verified: `useChat.sendMessage` early-returns at `useChat.ts:55` (`if (!threadId || isStreaming) return`) whenever `threadId` is null, so a message typed before any thread exists is dropped — no optimistic bubble, no network call, no feedback. `ChatPage` initializes `activeThreadId = null` and only sets it via `handleNewThread` or thread selection. The backend already supports everything the fix needs: `POST /api/threads` with `{}` creates an untitled thread (`ThreadCreate.title` defaults to `None`, verified `schemas.py:21-22`), and the message endpoint auto-titles from the first message when `title` is null (verified `chat.py:1121-1124`). No backend change is required.

The single hard technical problem is a **React stale-closure / state-batching race**. `useChat(activeThreadId)` is keyed on `activeThreadId`, and `sendMessage` captures `threadId` in its `useCallback` closure (deps `[threadId, isStreaming, showToast]`). Calling `setActiveThreadId(newId)` and then `sendMessage()` in the same handler does NOT help — the `sendMessage` reference in scope is still the previous render's closure, where `threadId === null`, so the guard at line 55 silently no-ops. A secondary hazard: changing `activeThreadId` re-runs the `loadMessages()` effect (`ChatPage.tsx:40-42`), which would clobber the optimistic messages just appended. The clean, idiomatic fix is to make `sendMessage` accept the thread id as a parameter (or read it from a ref) so it never depends on the stale closured value, and to ensure auto-create completes (and the optimistic bubble is appended) before any thread-id-driven `loadMessages()` can wipe it.

The visual layer is fully specified and approved in the UI-SPEC: extend the `{messages.length === 0 && (…)}` block at `ChatContainer.tsx:34-42` with a `text-2xl font-bold` headline, a `text-sm text-gray-400` sub-line, and 2–3 `<button>` chips (`bg-gray-800`/`hover:bg-gray-700` or bordered `border-gray-700`, never solid `bg-blue-600`), each ≥44px tall, calling `onSend(chipText)` directly. No new dependency, color, font, or size.

**Primary recommendation:** Place auto-create in `ChatPage.handleSend`: detect `activeThreadId === null`, `POST /api/threads {}` reusing the `handleNewThread` pattern, capture the new id, `setActiveThreadId(newId)`, and **pass that id explicitly into `sendMessage(content, { threadId: newId })`** so the send does not depend on the stale closure. Refactor `sendMessage` to accept an optional explicit thread id (falling back to the hook's `threadId`). Guard the `loadMessages()` effect against wiping optimistic state on the just-created thread.

### Pattern 1: Thread-id-as-parameter (the closure-proof send) — RECOMMENDED
**What:** Make `sendMessage` accept an optional explicit thread id, used in preference to the closured `threadId`. The auto-create branch passes the just-created id directly, so the send never reads stale state.
**When to use:** Always for this phase — it is the lowest-risk fix for the line-55 guard race.
**Example:**
```typescript
// useChat.ts — refactored signature (illustrative)
const sendMessage = useCallback(async (
  content: string,
  opts?: { retry?: boolean; threadId?: string }
) => {
  const effectiveThreadId = opts?.threadId ?? threadId   // explicit wins over stale closure
  if (!effectiveThreadId || isStreaming) return           // guard now passes for a fresh thread
  // ...use effectiveThreadId for the POST /api/threads/{effectiveThreadId}/messages URL...
}, [threadId, isStreaming, showToast])
```
```typescript
// ChatPage.tsx — handleSend with auto-create
const handleSend = async (content: string) => {
  let targetId = activeThreadId
  if (targetId === null) {
    const thread = await apiFetch('/api/threads', { method: 'POST', body: JSON.stringify({}) })
    targetId = thread.id
    setThreads(prev => [thread, ...prev])
    setActiveThreadId(thread.id)           // updates UI/sidebar; does NOT need to be awaited for the send
  }
  await sendMessage(content, { threadId: targetId })   // closure-proof
  loadThreads()                            // D-03: pick up backend auto-generated title
}
```
**Source:** Derived from verified codebase signatures (`useChat.ts:54-55`, `ChatPage.tsx:44-65`) + React stale-closure guidance [CITED: react.dev/reference/react-dom/flushSync].

### Anti-Patterns to Avoid
- **Calling `sendMessage()` immediately after `setActiveThreadId()` and expecting the new id:** the in-scope `sendMessage` is the prior render's closure (`threadId === null`); the guard at line 55 silently no-ops. This IS the original bug — re-introducing it by relying on the closure defeats the whole phase.
- **Putting the `POST /api/threads` create inside `useChat`:** the hook does not own `threads`/`setThreads` (that's `ChatPage`); creating there forces the new thread to be pushed to the sidebar via a callback prop, adding coupling. Keep create in `ChatPage.handleSend`.
- **Setting a placeholder `title: "New Chat"` on create:** violates D-03 and causes label flicker when the backend title arrives. Create with `{}` (title null).
- **Rendering chips as solid `bg-blue-600`:** reads as already-sent user bubbles and blows the 10% accent budget (UI-SPEC § Color, explicit prohibition).
- **Branching the entire empty-state layout on `isAnon`:** re-introduces the maintenance split the fix removes; unify the layout, gate only the optional D&D hint line on `isAnon` (UI-SPEC § Copywriting Contract).
- **`font-medium` on any new element:** the 2-weight contract allows only 400 and 600/700. (The existing Send button's `font-medium` at `ChatInput.tsx:40` is a known inherited violation — do NOT propagate it, do NOT need to fix it.)

**Key insight:** Almost nothing in this phase is *new behavior* — it is *unblocking existing behavior*. The optimistic bubble, the SSE stream, the create endpoint, the titling, and the title refresh all already work. The fix is to (a) let a null-thread send reach those mechanisms by establishing a thread id first, and (b) replace one `<p>` with a richer empty-state. Resist building anything net-new on the data path.

## Common Pitfalls

### Pitfall 1: Stale-closure guard silently swallows the first send
**What goes wrong:** After auto-create, `sendMessage()` (called with no explicit id) still sees `threadId === null` from its captured closure and returns at line 55 — the original silent no-op, now hidden one layer deeper.
**Why it happens:** `useChat(activeThreadId)` re-creates `sendMessage` only on the *next* render after `setActiveThreadId`. The function reference invoked synchronously in `handleSend` is the current (stale) one. React batches the state update; the new closure does not exist yet at call time.
**How to avoid:** Pass the freshly-created id explicitly: `sendMessage(content, { threadId: newId })` and have the guard use `opts.threadId ?? threadId` (Pattern 1). Never depend on `activeThreadId` having propagated.
**Warning signs:** First message after cold start / after delete does nothing; second message works (because by then `threadId` has updated). This exact "second-try works" symptom is the fingerprint of the stale closure.

### Pitfall 2: `loadMessages()` effect wipes the optimistic bubbles
**What goes wrong:** `setActiveThreadId(newId)` triggers the `useEffect(() => { loadMessages() }, [loadMessages])` at `ChatPage.tsx:40-42` (because `loadMessages` is keyed on `threadId`). For a brand-new thread, `loadMessages()` fetches `[]` and calls `setMessages([])`, erasing the optimistic user bubble + streaming placeholder that `sendMessage` just appended.
**Why it happens:** Two independent state flows race: the thread-switch effect (fetch + replace messages) and the send (append messages). Order is not guaranteed.
**How to avoid options (planner picks one, document the choice):**
  1. **Skip the reload for the just-created thread:** track a "just created, don't reload" flag/ref so the effect no-ops for the new id (its messages are already optimistic + streaming).
  2. **Make `loadMessages` non-destructive when already streaming / when messages exist for that id:** guard `if (isStreaming) return` or only replace when the fetched set is non-empty.
  3. **Don't switch `activeThreadId` until after the send establishes optimistic state** — but the sidebar then lags; less clean.
**Warning signs:** The user bubble flashes then disappears; the assistant streams into an empty pane; or messages double-render. Verify the empty-state's `{messages.length === 0}` block does not flicker back after the bubble appears.

### Pitfall 3: Title flicker from a placeholder
**What goes wrong:** Creating with `title: "New Chat"` shows "New Chat" in the sidebar/top bar, then it snaps to the real title after `loadThreads()`.
**Why it happens:** The top bar computes `activeThread?.title || (activeThreadId ? 'New Chat' : 'Chat')` (`ChatPage.tsx:68`) — a null title already falls back to "New Chat" *for display* without persisting it. Persisting a placeholder collides with the backend title.
**How to avoid:** Create with `{}` (title null). The existing fallback at line 68 already renders "New Chat" transiently for a null-title active thread, and `loadThreads()` replaces it. D-03 satisfied with zero new code.
**Warning signs:** Sidebar shows a literal "New Chat" entry that should have been the message-derived title.

### Pitfall 4: Empty-state doesn't vanish on first send (or chips remain tappable mid-stream)
**What goes wrong:** The empty-state lingers behind the first bubble, or chips stay clickable while streaming, allowing a double-send.
**Why it happens:** Empty-state visibility is `messages.length === 0`; if optimistic append is clobbered (Pitfall 2) it reappears. Chips have no `disabled` tie to `isStreaming`.
**How to avoid:** Keep the empty-state strictly inside `{messages.length === 0 && (…)}` (UI-SPEC requires this so it disappears the instant the optimistic bubble lands). Because the empty-state only renders when `messages.length === 0`, and the first send appends a user bubble synchronously, chips naturally disappear — but ensure Pitfall 2 doesn't resurrect them.
**Warning signs:** Two user bubbles from one chip tap; empty-state visible above messages.

### Pitfall 5: Create-failure path has no feedback
**What goes wrong:** If `POST /api/threads` throws (network/401), `handleSend` rejects before `sendMessage` runs — silent failure, the very class of bug this phase fixes.
**Why it happens:** `apiFetch` throws on non-2xx (`api.ts:35-38`); nothing catches it in the new branch.
**How to avoid:** Wrap the auto-create in try/catch; on failure, `Sentry.captureException(err)` (mirror `useChat.ts:216`) and `showToast(...)` the existing toast surface. The UI-SPEC says no NEW error copy is in scope, but the create path needs *some* feedback — reuse the existing toast mechanism rather than failing silently. Flag wording to the planner.
**Warning signs:** Tapping send/chip while offline does nothing visible.

## Open Questions
