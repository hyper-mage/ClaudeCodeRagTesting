<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OpenRouter redirect → browser callback | `code` + `state` arrive in the callback URL; `state` must be validated (OpenRouter omits it) |
| browser → Sentry vendor | events/breadcrumbs/URLs may carry a leaked `sk-or-v1-…` token |
| sessionStorage → exchange POST | `code_verifier` + `state` held same-tab; must not move to localStorage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-08 | Spoofing / Tampering (CSRF / callback session fixation) | OAuthCallbackPage | mitigate | SPA generates `state` (randomString) stored in sessionStorage; callback rejects when `returnedState !== storedState`. Acceptance criterion asserts the `!==` state check before the exchange POST. (Pitfall 2) |
| T-10-09 | Information Disclosure | lib/sentry.ts — `sk-or-v1-…` leaking to events/breadcrumbs/callback URL | mitigate | `/sk-or-v1-[A-Za-z0-9_-]+/g → '[redacted-key]'` applied in beforeSend (message, exception values, request.url) + beforeBreadcrumb (message, data). This is the frontend half of SEC-01. |
| T-10-10 | Information Disclosure | Raw exchange error / key rendered into the callback failure UI | mitigate | Failure copy is the hard-coded locked sentence; the caught error is never interpolated into the DOM (Pitfall 1 / D-06). Acceptance criterion greps for absence of `error.message`/`String(e)` in JSX. |
| T-10-11 | Information Disclosure / Tampering | `code_verifier`/`state` in localStorage (leaks across tabs) or lost on refresh (false failure) | mitigate | sessionStorage only (D-07 + CSRF binding); the callback reads from sessionStorage so hard-refresh re-runs successfully. Acceptance asserts sessionStorage (not localStorage) usage. (Pitfall 3) |
</threat_model>
