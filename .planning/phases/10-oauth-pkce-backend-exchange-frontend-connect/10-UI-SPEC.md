# Phase 10 — UI Design Contract

> Visual and interaction contract for Phase 10 (OAuth PKCE — Backend Exchange + Frontend Connect). Generated by gsd-ui-researcher, verified by gsd-ui-checker. This phase is **frontend-light BYOK connect**: a minimal `/settings` stub (connect / status / disconnect ONLY), an OAuth callback route with spinner→toast / inline-error states, an IconSidebar gear entry, and a connection-status dot in the chat header. It inherits the hardened token contract established by the v1.0/v1.1 UI-SPECs (most recently `08-portfolio-polish/08-UI-SPEC.md`) — **no new design language is introduced.**

---

## Scope (in / out)

**In scope (this contract — KEY-01, KEY-03, KEY-04):**
- **`/settings` stub** — a real, testable route showing key connection state ONLY: Connect CTA (not-connected) OR masked key tail + connected-since date + Disconnect (connected). (D-01)
- **IconSidebar gear entry** → navigates to `/settings`; mirrored into the `IconNavRow` drawer cluster for mobile. (D-02)
- **Chat-header connection dot** — a small connected/not-connected indicator, always visible at a glance. (D-02, D-04)
- **`/settings/openrouter/callback` route** — "Connecting your OpenRouter account…" spinner during `state`-validate → `POST exchange`; happy path auto-redirects to `/settings` (Connected) + success toast; failure path renders inline error + Retry / Back-to-settings. (D-05, D-06, D-07)
- **Disconnect confirm dialog** → `DELETE /api/keys` → flips to not-connected + Connect CTA. (D-08)
- **Reconnect = re-run Connect** (exchange upserts; one key per user — no separate "switch key" affordance). (D-09)

**Out of scope (deferred — do NOT design here):**
- Model picker / catalog — Phase 12 / 15.
- Balance display (`GET /api/keys/balance`), running cost — Phase 14 (COST-01/02/04).
- Default-model selection, theme toggle, full Settings/account page, profile — Phase 13 / 14 (PREF-01/02). The `/settings` route grows into the full page in Phase 14; this phase ships only the key-state block.
- The full "Demo mode vs Your key (balance $X) vs No key" state machine + mid-chat 401/402/403 recovery (Pitfall 13 full scope) — Phase 14. This phase ships ONLY the binary connected/not-connected dot.
- Key-gated model selection that launches OAuth inline + resume-on-return — Phase 15 (KEY-05).
- Manual API key paste — explicit milestone anti-feature (OAuth-only posture).

**Non-UI sibling work in this phase (mentioned for completeness, not specified by this contract):** the frontend Sentry `sk-or-v1-…` scrubber extension (`lib/sentry.ts`), the `lib/pkce.ts` Web Crypto helper, the backend `POST /api/keys/openrouter/exchange` / `GET /api/keys/status` / `DELETE /api/keys` endpoints, and the `openrouter_service.exchange_code` shape. **Security-critical UI invariant carried into this contract (D-06):** any surfaced failure reason MUST be scrubbed of `sk-or-…` before it reaches a log, Sentry event, toast, or rendered DOM node.

**Accent (`bg-blue-600`) reserved for, in this phase, exactly:**
1. The **"Connect OpenRouter"** primary CTA on the `/settings` not-connected state.
2. The **"Retry"** button on the callback failure state (so the recovery path reads as primary).

It is **NOT** used for: the gear-nav active state (stays `bg-gray-800`, matching Chat/Documents), the connection dot (green/gray, never blue), the Disconnect button (red), the "Back to settings" secondary link (neutral gray), or any decorative element.

**Destructive (`bg-red-*`) reserved for, in this phase, exactly:**
1. The **Disconnect** confirm button inside `ConfirmDialog` (existing `bg-red-600 hover:bg-red-700` — Disconnect stops chat, so it is genuinely destructive).
2. The inline **callback failure** sentence (`text-red-400`) + its leading `AlertCircle` icon.

It is **NOT** used for the not-connected status (that is neutral gray — an empty state, not a failure).

**Success (`green-*`) reserved for, in this phase, exactly:**
1. The **connected** status dot (`bg-green-500`) in the chat header and on `/settings`.

## Copywriting Contract

All copy is **specific, non-technical, and free of HTTP codes**; the provider name "OpenRouter" IS used here (it is the literal account being connected — unlike the generic chat-error copy that hides provider names). Failure copy must never echo a raw error or any `sk-or-…` fragment (D-06).

| Element | Copy | Notes |
|---------|------|-------|
| Primary CTA (not-connected, `/settings`) | **Connect OpenRouter** | Verb + noun. Launches the PKCE redirect to OpenRouter `/auth`. |
| Not-connected — explainer (Label/Body under CTA) | Connect your OpenRouter account to chat with your own key. No key to copy or paste. | Sets the OAuth-only, zero-paste expectation. One line. |
| Connected — status line | Connected | `text-green-400`, next to the green dot. |
| Connected — masked key (Body) | `sk-or-v1-…wXyZ` | Last-4 masked tail (D-03). The literal middle is the single-char ellipsis `…` (not three dots). Rendered from the stored non-secret display label, never reconstructed from ciphertext. |
| Connected — connected-since (Label) | Connected since {Mon D, YYYY} | e.g. "Connected since Jun 19, 2026". Sourced from the stored `connected_at`/`connected_since` column. Format is locale-friendly short date; no time-of-day. |
| Section label (`/settings`) | OpenRouter | `text-base font-semibold` heading for the key-state block. |
| Disconnect — button label | Disconnect | Single word, red. |
| Callback — in-flight (spinner caption) | Connecting your OpenRouter account… | Trailing single-char ellipsis `…`. Shown while `state`-validate → `POST exchange` is in flight. |
| Callback — success | *(no dedicated screen)* auto-redirect to `/settings` + success toast | Happy path shows no extra click (D-05). |
| Connect success — toast body | OpenRouter connected. | `'success'` variant, auto-dismiss 4s. Fires after redirect lands on `/settings` (Connected). |
| Callback — failure (inline, `text-red-400`) | Couldn't connect your OpenRouter account — please try again. | No HTTP code, no raw reason, no `sk-or-…`. Covers forged/mismatched CSRF `state`, exchange 403, and network error identically (D-06). |
| Callback — failure primary action | Retry | `bg-blue-600`. Re-runs the connect flow (fresh PKCE init). |
| Callback — failure secondary action | Back to settings | Neutral gray link/button → navigates to `/settings`. |
| Disconnect confirmation — heading | Disconnect OpenRouter? | ConfirmDialog `heading`. |
| Disconnect confirmation — body | You'll need to reconnect to chat with your own key. | ConfirmDialog `body` (locked verbatim by D-08). |
| Disconnect confirmation — confirm label | Disconnect | ConfirmDialog `confirmLabel`, red button. |
| Disconnect confirmation — cancel label | Cancel | ConfirmDialog `cancelLabel`, gray button. |

**Empty state (`/settings` with no connected key):** This IS the not-connected state above — the "Connect OpenRouter" CTA + explainer line. There is no separate "no data" screen; the not-connected state is the empty state.

**Destructive actions in this phase (and confirmation approach):**

| Action | Confirmation | Copy |
|--------|--------------|------|
| **Disconnect OpenRouter** (`DELETE /api/keys`) | `ConfirmDialog` (existing portal component) — Disconnect stops chat because demo-fallback is OFF by default, so it is genuinely destructive and MUST confirm (D-08). | heading "Disconnect OpenRouter?" / body "You'll need to reconnect to chat with your own key." / confirm "Disconnect" (red) / cancel "Cancel" |

Reconnect is NOT destructive (it upserts and overwrites the prior row by design, D-09) — no confirmation; it simply re-runs Connect.

## Surface 1 — `/settings` Stub (KEY-03, KEY-04)

**Location:** new route `/settings` in `frontend/src/App.tsx`; new page e.g. `frontend/src/pages/SettingsPage.tsx`. Wrapped in the same authenticated shell as `/` and `/documents` (IconSidebar + content), protected by the existing `ProtectedRoute`.

**Layout shell:** reuse the app's existing left rail (IconSidebar on desktop, MobileTopBar + drawer on mobile). The route renders a single content panel — a `bg-gray-900` card (`rounded-lg p-6`) on the `bg-gray-950` page, `max-w-md` so it reads as a focused settings block, not a full-bleed page (this leaves room for Phase 14 to add more sections below).

**Two states (driven by `GET /api/keys/status` → `{ connected: boolean, masked_label?: string, connected_at?: string }`):**

```
NOT CONNECTED                              CONNECTED
┌──────────────────────────────┐          ┌──────────────────────────────┐
│ Settings            (Display) │          │ Settings            (Display) │
│                               │          │                               │
│ OpenRouter         (Heading)  │          │ OpenRouter         (Heading)  │
│ Connect your OpenRouter       │          │ ● Connected         (green)   │
│ account to chat with your     │          │ sk-or-v1-…wXyZ      (Body)    │
│ own key. No key to copy or    │          │ Connected since Jun 19, 2026  │
│ paste.            (Body/Label)│          │                     (Label)   │
│                               │          │                               │
│ [ Connect OpenRouter ]        │          │ [ Disconnect ]      (red)     │
│   (bg-blue-600, full-width)   │          │   (bg-red-600)                │
└──────────────────────────────┘          └──────────────────────────────┘
```

**Interaction:**
- **Connect OpenRouter** → run PKCE init (`lib/pkce.ts`: generate `code_verifier` + S256 `code_challenge` + CSRF `state`, persist verifier+state to `sessionStorage`), then `window.location.assign(...)` to OpenRouter `/auth` with `callback_url = window.location.origin + '/settings/openrouter/callback'`.
- **Disconnect** → open `ConfirmDialog` → on confirm, `DELETE /api/keys` via `apiFetch` → on success, refetch status (or optimistically flip to not-connected) → render the not-connected state.
- **Loading:** while `GET /api/keys/status` is in flight on first mount, show a neutral inline spinner (same `animate-spin` pattern) or a muted skeleton line — do NOT flash the not-connected CTA before status resolves (avoids a connect→disconnect flicker on reload for already-connected users).

**Mobile (≤768px):** the `max-w-md` card sits full-width within the content column under the `MobileTopBar`. Connect / Disconnect buttons are full-width, `py-3` (≥44px). Reached via the gear entry in the mobile drawer (`IconNavRow`).

## Surface 3 — Chat-Header Connection Dot (KEY-03, D-02 / D-04)

**Purpose:** key state is always visible at a glance without opening `/settings` (D-02). This phase ships ONLY a binary dot — the full "Demo / Your key / No key" state machine is Phase 14 (deferred).

**Treatment:** an 8px round dot (`h-2 w-2 rounded-full`) plus an optional adjacent label:
- **Connected:** `bg-green-500`, optional label "Connected" (`text-green-400 text-xs`). `aria-label="OpenRouter connected"`.
- **Not connected:** `bg-gray-500`, optional label "Not connected" (`text-gray-400 text-xs`). `aria-label="OpenRouter not connected"`.
- The dot is **non-interactive** (display-only) per D-04, OR may optionally be wrapped in a button that navigates to `/settings` (planner discretion — if interactive, it must clear the 44px touch target on mobile; if display-only, no size floor). Default recommendation: display-only dot, since the gear entry already provides the `/settings` route.

**Location:** the chat surface's header region. The desktop chat (`ChatContainer` / `ChatPage`) currently has no dedicated header bar — the dot should be placed where the active-thread title reads (desktop: top-of-`ChatContainer` or the `ThreadSidebar`/header seam; **mobile: the `MobileTopBar`**, to the LEFT of the existing `DemoPill` right-slot so demo-identity and key-state read as two distinct signals). Exact desktop anchor is planner discretion within the existing chat shell — the contract requirement is: the dot is visible on the chat route at all times, distinct from the `DemoPill`, and never collides with it.

**Data source:** `GET /api/keys/status` `.connected`. Fetch once on chat mount; refetch after a successful connect/disconnect so the dot stays accurate within a session (a lightweight shared hook, e.g. `useKeyStatus`, is the natural home — planner decides). Do NOT poll.

## Surface 4 — `/settings/openrouter/callback` Route (KEY-01)

**Location:** new route `/settings/openrouter/callback` in `frontend/src/App.tsx`; new page e.g. `frontend/src/pages/OAuthCallbackPage.tsx`. Served by the Cloudflare Pages SPA fallback in prod (Pitfall 6 — non-UI infra concern, noted). Route guard (public vs protected) is planner discretion per CONTEXT.md.

**Three visual states:**

```
IN-FLIGHT (default on mount)        SUCCESS (transient)            FAILURE
┌──────────────────────────┐        auto-redirect → /settings     ┌──────────────────────────┐
│        (centered)        │        + success toast               │        (centered)        │
│         ◯  (spinner)     │        "OpenRouter connected."        │   ⚠  (AlertCircle, red)  │
│  Connecting your         │        (no dedicated screen)         │  Couldn't connect your   │
│  OpenRouter account…     │                                      │  OpenRouter account —    │
│        (Body)            │                                      │  please try again.       │
│                          │                                      │     (text-red-400, Body) │
└──────────────────────────┘                                      │  [ Retry ]  Back to      │
                                                                   │  (blue)     settings     │
                                                                   └──────────────────────────┘
```

**In-flight:** centered card (`bg-gray-900 rounded-lg p-6 max-w-sm` on `bg-gray-950`), the existing `animate-spin` ring spinner (`border-2 border-gray-400 border-t-transparent rounded-full`, ~24px) above the caption "Connecting your OpenRouter account…". This is the state during `state`-validate → `POST /api/keys/openrouter/exchange`.

**Flow on mount:** read `code` + `state` from the URL query; read the stored `code_verifier` + `state` from `sessionStorage`; validate the returned `state` matches the stored one (CSRF check — OpenRouter omits `state`, the SPA owns it). If valid → `POST {code, code_verifier}` to the backend exchange endpoint via `apiFetch`. On 2xx → `navigate('/settings', { replace: true })` and fire `showToast('OpenRouter connected.', 'success')`. On `state` mismatch, exchange non-2xx (e.g. 403), or network error → render the **failure** state.

**Hard-refresh is the SUCCESS path (D-07):** `code_verifier` + `state` live in `sessionStorage` (survive same-tab refresh), so a refreshed callback re-runs the exchange successfully. Do NOT treat a refresh as a failure or show an error on reload.

**Failure state (D-06):**
- `AlertCircle` lucide icon (`size={24}`, `text-red-400`) above the locked sentence "Couldn't connect your OpenRouter account — please try again." (`text-sm text-red-400`, line-height 1.5).
- **Retry** (`bg-blue-600 hover:bg-blue-700`, primary) → re-runs the full connect flow (fresh PKCE init + redirect). 
- **Back to settings** (neutral `bg-gray-700 hover:bg-gray-600` or a plain `text-gray-400 hover:text-white` link) → `navigate('/settings')`.
- The surfaced reason MUST be the generic locked sentence only — never the raw exchange error, HTTP status, or any `sk-or-…` fragment. The Sentry scrubber extension is the backstop; the UI copy is hard-coded and never interpolates a server message.

**Mobile:** the centered card is `max-w-sm` full-width-within-viewport with `mx-4` gutters (matches `ConfirmDialog` sizing). Retry / Back buttons stack or sit side-by-side; each clears 44px (`py-3`).

## Accessibility

- **Connect / Disconnect / Retry / Back:** native `<button type="button">`; default Tailwind-4 focus-visible ring. Visible labels are unambiguous — no `aria-label` needed.
- **Connection dot:** if display-only, the colored dot alone is insufficient for screen readers — pair it with an `aria-label` on its wrapper (`"OpenRouter connected"` / `"OpenRouter not connected"`) and a visible text label where space allows. Never rely on color alone to convey state (color-blind safe: green vs gray differ in lightness; the text label is the source of truth).
- **Callback in-flight:** wrap the spinner+caption region in `role="status" aria-live="polite"` so the "Connecting…" state is announced; announce the failure sentence via `role="alert"`.
- **ConfirmDialog (Disconnect):** reused component already handles Escape-to-cancel and click-outside; ensure focus moves into the dialog on open and the confirm button is reachable (the existing portal renders to `document.body`).
- **Success toast:** existing `ToastProvider` declares `aria-live="polite" aria-atomic="true"` + `role="status"` — reuse as-is.
- **Keyboard:** the gear nav entry, Connect, Disconnect, Retry, and Back are all Tab-focusable real buttons. The masked-key line and connected-since line are static text (not focusable).
- **Contrast:** all combinations clear WCAG AA (see Color § contrast note).
