<objective>
Resolve T-08-03 (retry creates duplicate assistant row in DB). This is the backend half of PORT-02 — the frontend will append `?retry=true` to the POST when the user clicks the Retry button in the error bubble. Without this backend fix, a successful retry leaves an orphan empty assistant bubble above the new answer on thread reload, which looks broken.

Per RESEARCH Pitfall 3 Strategy A (recommended), the backend on retry:
1. Deletes the most-recent assistant row for the thread (the one with empty/failed content).
2. SKIPS the user-message insert at chat.py:491 (the original user message is preserved from the original failed send).
3. Proceeds with the normal LLM call + assistant placeholder insert + SSE stream.

Purpose: Make chat retry idempotent at the data layer — one user message + one final assistant message per "logical send", regardless of how many failed attempts preceded it.
Output: Modified `send_message` handler in chat.py + 3 passing tests in test_chat_retry.py.
</objective>
