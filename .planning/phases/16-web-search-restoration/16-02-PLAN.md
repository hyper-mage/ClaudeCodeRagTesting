<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend → Tavily (api.tavily.com) | The owner `tvly-` key crosses here in an Authorization header over HTTPS; it must never cross back toward the client or into logs. |
| Tavily error / exception → SSE + logs | `str(e)` / `exc_info=True` from a failed search could echo request details out of the trust boundary. |
| tool result → frontend tool card | The mapped `{answer, results}` / `{error}` payload (never the key or the raw request) is what reaches the browser via the tool_result SSE event. |
| LLM tool-arg `query` → Tavily POST body | The query is a JSON body param (parameterized), not concatenated into SQL/HTML/a URL — no injection/SSRF sink. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-03 | Information Disclosure | tvly- key in logs via exc_info=True | mitigate | Key rides the Authorization header only, never an interpolated string; `_search_tavily`'s `logger.error(str(e), exc_info=True)` echoes method+URL, not the header. Defense-in-depth: scrub_secrets extended to redact `tvly-` (Task 3), covering the whole-process _ScrubFilter log path. test_scrub_secrets_redacts_tavily asserts redaction. |
| T-16-04 | Information Disclosure | tvly- key in the SSE error / tool-result payload | mitigate | The returned dict is `{"error": str(e), "results": []}` with the key absent from `str(e)`; the tool_result SSE event carries only the output preview + is_error flag. No key path to the browser. |
| T-16-05 | Information Disclosure | owner key sent to the browser | accept→mitigated | Tool executes server-side only; the frontend receives the mapped results / error, never the key or the request (verified: execute_tool → search_web server path). |
| T-16-06 | Tampering / SSRF | user-controlled search URL | accept | Endpoint is the hardcoded constant `https://api.tavily.com/search`; only the `query` string is variable — no user-controlled URL, no SSRF surface. |
| T-16-07 | Tampering | TLS downgrade / MITM | accept | Fixed `https://` endpoint; httpx verifies certs by default; `verify=False` is NOT introduced. |
| T-16-08 | Denial of Service | slow provider blocks the async SSE loop | accept | Bounded `timeout=30` on httpx.post; a timeout degrades to the graceful `{"error"}` path (WSRCH-04). Re-architecting the sync-in-async tool call is out of scope for a restoration (RESEARCH Pitfall 5). |
| T-16-09 | Spoofing / Elevation (cost) | tool exposed without a configured key | mitigate | Fail-closed preserved: `settings.web_search_enabled` gates the tool append AND search_web re-guards; verified by inspection + test_gating_fail_closed. No per-user keys (owner-configured server tool only). |
