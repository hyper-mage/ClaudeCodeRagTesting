# Phase 999.1: Code Review Report

**Reviewed:** 2026-06-24T17:13:27Z
**Depth:** standard
**Files Reviewed:** 9
**Status:** issues_found

## Summary

Phase 999.1 is a frontend-only UX change: auto-create a thread on first send from a
null-thread state, render an optimistic user bubble, and show a welcoming empty-state
with example-prompt chips. The implementation is well-commented and the test suite is
unusually thorough for the *covered* paths (closure-proof `threadId`, no-clobber-while-
streaming, the stream-end refetch regression). The core auto-create-on-send chain is sound.

The review focused, per the phase brief, on the React effect/ref/closure interplay around
`loadMessages`, `isStreamingRef`, `skipNextLoadRef`, and abort handling — and that is exactly
where the uncovered defects live. The single most important finding (CR-01) is a cross-thread
state-clobber: switching to a *different existing* thread while a stream is in flight does not
abort the stream and does not load the target thread's messages, leaving one thread's streamed
content rendered under another thread's header. The tests never switch threads mid-stream, so
this is invisible to the suite. Several warnings concern the `isStreaming` re-entrancy guard,
the swallowed-abort placeholder, the `skipNextLoadRef` one-shot assumption, and a silent-on-load-
error behavior that masks real failures behind the empty state.

## Critical Issues

### CR-01: Switching to another thread mid-stream bleeds streamed content across threads and never loads the target thread

**File:** `frontend/src/hooks/useChat.ts:46-65`, interacting with `frontend/src/pages/ChatPage.tsx:50-57,127,142`

**Issue:**
While a send is in flight in thread A (`isStreamingRef.current === true`), the user can click
thread B in the sidebar (`onSelectThread={setActiveThreadId}`, ChatPage:127 / drawer:142). That
fires `setActiveThreadId(B)` → `useChat(B)` re-renders → `loadMessages` identity changes (dep
`[threadId]`) → the load effect (ChatPage:50-57) runs `loadMessages()`. But `loadMessages`
returns early on `if (isStreamingRef.current) return` (useChat.ts:55), so **thread B's messages
are never fetched**. Meanwhile nothing aborts thread A's stream — `cancel()` only runs on unmount
(ChatPage:37-39) and `abortRef` is only aborted by the *next send* (useChat.ts:94). The in-flight
stream keeps appending content deltas into the shared `messages` array (keyed by `assistantId`,
not by thread), so thread A's reply renders under thread B's header. When the stream finishes,
`finally` clears `isStreamingRef`, but the load effect for B does **not** re-run (its dep
`threadId`/`loadMessages` identity did not change again), so B is left displaying A's content
until the next unrelated thread switch.

This is the same class of bug the `isStreamingRef` change was meant to fix (post-stream clobber),
but the ref guard only *defers* the load — it never schedules the load that was skipped. The
no-clobber guard protects the optimistic bubble at the cost of silently dropping a legitimate
thread switch. Result: wrong-thread content (a correctness/data-integrity defect the user will
read as "my message went to the wrong conversation").

The test suite does not cover any mid-stream thread switch (ChatPage Test 7 only asserts the
*same* auto-created thread is not refetched; useChat Test 4 calls `loadMessages` manually but
never changes `threadId`), so this ships green.

**Fix:**
Abort the in-flight stream on thread switch and re-load once streaming actually stops. Two
coordinated changes:

```ts
// useChat.ts — re-run the load when a thread switch was suppressed mid-stream.
// Track whether a load was skipped due to streaming, and flush it when the stream ends.
const pendingLoadRef = useRef(false)

const loadMessages = useCallback(async () => {
  if (!threadId) { setMessages([]); return }
  if (isStreamingRef.current) { pendingLoadRef.current = true; return }
  // ...existing fetch...
}, [threadId])

// In sendMessage `finally`, after clearing isStreamingRef:
if (pendingLoadRef.current) {
  pendingLoadRef.current = false
  void loadMessages()
}
```

Better still: when `threadId` changes while a stream is active, *abort* it first so the user's
new context is authoritative. Expose the active-thread identity to the stream and have the
load effect call `cancel()` before switching, or capture the streaming thread id and discard
deltas whose target thread no longer matches `activeThreadId`. At minimum, add a test that
selects a second thread mid-stream and asserts the second thread's messages load.

## Warnings

### WR-01: `sendMessage` re-entrancy guard reads a stale `isStreaming`, allowing a double-send race

**File:** `frontend/src/hooks/useChat.ts:72`

**Issue:**
The guard `if (!effectiveThreadId || isStreaming) return` reads `isStreaming` from the closure
(state, recreated only when `[threadId, isStreaming, showToast]` change — useChat.ts:261). On the
very first send from a null-thread state, `handleSend` (ChatPage:108) calls `sendMessage` with the
closure captured *before* `isStreaming` flips. Two near-simultaneous triggers — e.g. a fast double
Enter, or a chip tap immediately followed by a typed Enter — can both pass the guard because the
state-based `isStreaming` has not yet propagated to a new `sendMessage` identity. `abortRef.current?.abort()`
(useChat.ts:94) then aborts the first request, and two optimistic assistant placeholders are
appended (useChat.ts:89). The deliberate choice everywhere else in this code is to use a ref to
defeat exactly this stale-closure timing (`isStreamingRef`), but the entry guard still uses state.
`ChatInput` disables the textarea while streaming, but the empty-state chips (ChatContainer:52-60)
are not disabled, and `isStreaming` lags a render behind, so the chip path is the most exposed.

**Fix:** Guard on the ref, consistent with the rest of the hook:

```ts
if (!effectiveThreadId || isStreamingRef.current) return
// then immediately:
isStreamingRef.current = true
```

Set `isStreamingRef.current = true` synchronously at the top of the guarded section (before the
first `await`) so the second concurrent call short-circuits.

### WR-02: Aborted send leaves a permanent empty assistant placeholder bubble

**File:** `frontend/src/hooks/useChat.ts:226-230, 88-89`

**Issue:**
On `AbortError` the catch returns early (useChat.ts:227-229) without removing the empty assistant
placeholder appended at useChat.ts:89 (`{ id: assistantId, role: 'assistant', content: '' }`).
Aborts happen on (a) a subsequent send aborting the prior in-flight one (useChat.ts:94) and (b)
unmount via `cancel()`. In case (a), the prior turn's blank assistant bubble is left in `messages`
with empty content — it renders as an empty `MessageBubble`. Because `messages.length` is now > 0,
the empty-state also never returns. The comment claims "tool events already persisted by backend",
but the *client* placeholder is not reconciled. This is a visible dangling-empty-bubble artifact
whenever two sends overlap (which WR-01 makes reachable).

**Fix:** On abort, strip the placeholder before returning:

```ts
if (err instanceof DOMException && err.name === 'AbortError') {
  setMessages(prev => prev.filter(m => m.id !== assistantId))
  return
}
```

### WR-03: `skipNextLoadRef` assumes exactly one intervening load — a second render between create and effect leaks the clobber

**File:** `frontend/src/pages/ChatPage.tsx:50-57, 91-94`

**Issue:**
`skipNextLoadRef` is set true (ChatPage:93) right before `setActiveThreadId(thread.id)` (ChatPage:94),
on the assumption that the *next* run of the load effect (ChatPage:50-57) is the clobbering one and
consuming the flag once is sufficient. That holds only if the effect's next fire corresponds to the
new-thread load. But `sendMessage` (called immediately after at ChatPage:108) sets `isStreaming`
state, which re-creates `loadMessages`' consumers' identities elsewhere; more importantly, if any
state update causes the load effect to run *before* `activeThreadId` commits (e.g. a batched
re-render ordering under React 19 concurrent features, or a future `loadThreads` that also touches
state in the same tick), the one-shot flag is consumed by the wrong run and the real new-thread load
proceeds and clobbers the optimistic bubble. The code leans on two overlapping guards
(`skipNextLoadRef` here + `isStreamingRef` in the hook) precisely because neither is individually
airtight; relying on consume-exactly-once is fragile. Today it works because `isStreamingRef` is the
real backstop — but then `skipNextLoadRef` is redundant complexity that can desync.

**Fix:** Make the suppression keyed to the thread it was meant for, not a bare boolean, so a
mismatched effect run cannot consume it:

```ts
const skipLoadForRef = useRef<string | null>(null)
// on create:
skipLoadForRef.current = thread.id
// in effect:
if (skipLoadForRef.current === activeThreadId) { skipLoadForRef.current = null; return }
```

(Requires the effect to also depend on/read `activeThreadId`.) Alternatively, drop
`skipNextLoadRef` entirely and rely on the hook's streaming guard, which is the actual load
blocker — removing the redundant path removes the desync surface.

### WR-04: `loadMessages` swallows all fetch errors, hiding real failures behind the empty state

**File:** `frontend/src/hooks/useChat.ts:56-64`

**Issue:**
The `catch {}` block (useChat.ts:62-64) silently discards every error from
`apiFetch('/api/threads/${threadId}')`, including 401/403/500 and network failures. The comment
says this preserves prior "silent-on-error" behavior, but the prior behavior was `if (res.ok)` —
which only skipped the state update on non-2xx; it did not also swallow thrown network errors with
zero observability. With this change, a failed thread load leaves `messages` at its previous value
(often `[]`), so the user sees the welcoming empty-state ("Ask about any board game") for a thread
that actually has content and simply failed to load — a misleading success-looking state. There is
no `Sentry.captureException` here either, unlike every other error path in this file (useChat.ts:234,
ChatPage:99), so these failures are completely invisible in production.

**Fix:** Distinguish "empty thread" from "load failed". At minimum capture to Sentry; ideally
surface a non-destructive retry affordance:

```ts
} catch (err) {
  Sentry.captureException(err)
  // leave messages intact; optionally set a load-error flag for the UI
}
```

### WR-05: `retryLastUserMessage` can re-send to the wrong (or null) thread after an auto-create

**File:** `frontend/src/hooks/useChat.ts:272-278`

**Issue:**
`retryLastUserMessage` calls `sendMessage(lastUser.content, { retry: true })` with **no explicit
`threadId`** (useChat.ts:277), so `effectiveThreadId` falls back to the closured `threadId`
(useChat.ts:71). The whole point of Pitfall 1 / the explicit-`threadId` plumbing is that the
closured value is stale immediately after an auto-create. If the very first send auto-creates a
thread and then *fails* (network drop during stream), the error bubble appears; if the user taps
Retry before `activeThreadId`'s state update has propagated into a new `useChat` closure, retry can
fire against a stale/`null` `threadId` and silently no-op (the guard at useChat.ts:72 returns).
This is the exact failure mode the explicit-threadId design exists to prevent, reintroduced on the
retry path. The window is narrow (state usually commits first), but it is real and untested — no
retry-after-auto-create test exists.

**Fix:** Thread the active id through retry the same way `handleSend` does. Either capture the id
the failed send actually used and pass it back in, or have `ChatPage` own retry and pass
`{ threadId: activeThreadId }`. At minimum, document that retry depends on `activeThreadId` having
committed and add a test for retry immediately following an auto-create.

## Info

### IN-01: Duplicated `Message`/`ToolEvent` type definitions drift between hook and component

**File:** `frontend/src/components/ChatContainer.tsx:6-20` vs `frontend/src/hooks/useChat.ts:15-30`

**Issue:** `ChatContainer` redeclares local `ToolEvent` and `Message` interfaces instead of importing
the exported ones from `useChat`. They have already drifted: the hook's `ToolEvent` has `subEvents?`
and the hook's `Message` is the source of truth, but the component's copies omit `subEvents`. Passing
`msg.toolsUsed` (which may carry `subEvents`) into a `MessageBubble` typed against the narrower local
copy risks silent shape mismatches as the SSE schema evolves.

**Fix:** `import type { Message, ToolEvent } from '../hooks/useChat'` and delete the local copies.

### IN-02: `onSend` typed as fire-and-forget `() => void` while `handleSend` is async

**File:** `frontend/src/components/ChatContainer.tsx:24` / `frontend/src/pages/ChatPage.tsx:76,133`

**Issue:** `Props.onSend: (content: string) => void` but `handleSend` returns a `Promise`. The
chip `onClick={() => onSend(q)}` (ChatContainer:55) and `ChatInput` both ignore the returned promise,
so any rejection inside `handleSend` before its own try/catch (there is none currently, but future
edits) would become an unhandled rejection. Cosmetic today; brittle as the handler grows.

**Fix:** Type as `(content: string) => void | Promise<void>` and ensure all async branches are
internally caught (they currently are, via the create try/catch and `sendMessage`'s own catch).

### IN-03: Empty-state chips are not disabled while streaming, unlike the composer

**File:** `frontend/src/components/ChatContainer.tsx:50-61`

**Issue:** `ChatInput` receives `disabled={isStreaming}` (ChatContainer:83), but the example-prompt
chips have no disabled state. They only render while `messages.length === 0`, so in practice a stream
hides them — but the optimistic user bubble appends synchronously, so the window is effectively closed.
Still, decoupling the chips' enabled state from the same `isStreaming` signal the composer uses is an
inconsistency that feeds the WR-01 double-send window.

**Fix:** Pass `isStreaming` down and add `disabled={isStreaming}` to the chip buttons (with matching
`disabled:opacity-50`), or rely on the messages-length gate explicitly.

### IN-04: Stale comment references line numbers that no longer match

**File:** `frontend/src/test/setup.ts:11`

**Issue:** The comment "useChat.ts calls crypto.randomUUID() ... (lines 63/71)" points at lines that
do not match the current `useChat.ts` (the calls are at useChat.ts:80 and :88). Minor, but these
hard-coded line references rot quickly and mislead the next reader.

**Fix:** Reference the function names (`sendMessage` optimistic-id generation) instead of line numbers.
