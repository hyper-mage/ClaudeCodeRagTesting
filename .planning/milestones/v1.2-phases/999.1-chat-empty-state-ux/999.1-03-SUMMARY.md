# Phase 999.1 Plan 03: Auto-Create-on-Send Wiring Summary

**Wired auto-create-on-send into `ChatPage.handleSend` — a message sent against a null thread (cold start OR post-delete) now silently POSTs `/api/threads {}`, sends into the server-issued id with an explicit `threadId` (defeating the stale closure), renders the optimistic user bubble, and refreshes titles via `loadThreads()`; create failures surface a toast + Sentry. A live human-verify checkpoint confirmed the deliverable and surfaced a post-stream refetch-clobber bug, fixed in `ad43b9a`.**

- **Completed:** 2026-06-24

## Accomplishments
- **D-01 / D-04 auto-create-on-send:** `handleSend` now computes `targetId = activeThreadId`; if `null`, it `await apiFetch('/api/threads', { method:'POST', body: JSON.stringify({}) })`, sets `targetId = thread.id`, updates the sidebar (`setThreads` + `setActiveThreadId`), then sends. The same branch fires for both the cold-start and post-delete null states — one code path.
- **Closure-proof send:** `await sendMessage(content, { threadId: targetId })` passes the freshly-created id by value, so the optimistic user bubble renders on the very first send instead of silently no-opping (the dropped-first-message root cause — RESEARCH Pitfall 1). No `flushSync`, no create-in-hook.
- **D-03 untitled create + title refresh:** the create body is exactly `{}` (no `title`), so the backend titles the thread from the first message; `loadThreads()` after the send surfaces that title — no placeholder "New Chat" is persisted.
- **Create-failure feedback (Pitfall 5):** a rejected create calls `Sentry.captureException(err)` + `showToast(<existing generic copy>, 'error')` and **returns before `sendMessage`** — no silent dead-end; the toast copy never interpolates the caught error / HTTP status / response body (T-999.1-08).
- **Optimistic-bubble survival, two windows:** `skipNextLoadRef` suppresses the single `loadMessages` that the post-create `setActiveThreadId` triggers (the new thread has no server messages — that load is pure clobber); the checkpoint fix additionally moved the in-stream guard to `isStreamingRef` and depends `loadMessages` on `[threadId]` only, so a stream **ending** no longer re-fires the load and refetch-clobbers the streamed reply.
- **7 jsdom tests** (`ChatPage.test.tsx`) drive the real `onSend → handleSend → sendMessage` chain: cold-start auto-create + visible bubble + `loadThreads`, chip-tap parity, post-delete parity, untitled-`{}` body, create-failure (no `apiStream`, toast + Sentry), anon-hint page parity, and the stream-end no-clobber regression.

## Decisions Made
- **Explicit-id parameter passing** is the closure-proof fix (RESEARCH "Don't Hand-Roll") — `flushSync` and moving the create into `useChat` were both explicitly avoided.
- **Reused generic toast copy** for create failures (`"The assistant didn't respond. Tap the message to retry."`) rather than inventing create-specific copy — keeps the locked generic-failure rule (T-999.1-08) intact. *Flag for future polish:* a create-specific string ("Couldn't start a new chat — try again.") would read marginally better, but shipping a reused generic string is the safer call and matches the phase's no-new-error-copy guidance.
- **Added `skipNextLoadRef`** alongside Plan 01's `loadMessages` guard — the jsdom test proved the post-create `setActiveThreadId` re-ran the load effect and clobbered the optimistic bubble, so a deterministic one-shot skip was warranted (the plan permitted this "if the test proves a clobber remains").
- **Checkpoint fix `ad43b9a`** (decided live during human-verify): the in-stream guard was protecting only DURING the stream; Plan 01's adding `isStreaming` to `loadMessages` deps made the thread-load effect re-fire when streaming STOPPED, refetching `/api/threads/{id}` and clobbering the streamed reply / error bubble the instant the send finished. Moved the guard to a ref and reduced the deps to `[threadId]`.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Post-stream refetch clobbered the streamed reply / error bubble**
- **Found during:** Task 3 (live human-verify checkpoint)
- **Issue:** Plan 01 added `isStreaming` to `loadMessages`' dependency array so the in-stream guard could read it. That also made the ChatPage thread-load effect re-fire whenever streaming **stopped** — refetching `/api/threads/{id}` and replacing the just-streamed messages (and any error bubble) the instant the send finished. jsdom masked it because `findByText` resolved in the transient pre-clobber window; only the live stack exposed the flash-then-vanish.
- **Fix:** Moved the in-stream guard to `isStreamingRef` (a ref) and reduced `loadMessages`' deps to `[threadId]` only, so the load effect fires solely on thread change, not on stream-end. Added `ChatPage.test.tsx` Test 7 as a regression (asserts the streamed reply survives stream-end with no post-stream refetch clobber).
