# Phase 999.1 — UI Design Contract

> Visual and interaction contract for Phase 999.1 (Chat empty-state UX). Generated by gsd-ui-researcher. This phase is a **frontend-only UX fix**: make the first chat message "just work" by auto-creating a thread on send (D-01), and replace the bland text-only empty-state at `ChatContainer.tsx:34-42` with a welcoming headline + 2–3 tappable board-game example prompts (D-02). It inherits the hardened, dark-only manual-Tailwind token contract established by `10-UI-SPEC.md` (and the v1.0/v1.1 chain before it) — **no new design language, no new dependency, no new color, no new font/weight/size is introduced.**

---

## Scope (in / out)

**In scope (this contract — chat empty-state UX):**
- **Welcoming empty-state in the message pane** — when `messages.length === 0`, render a short headline + sub-line + 2–3 tappable example-prompt chips, replacing the current text-only `<p>` at `ChatContainer.tsx:34-42` (D-02).
- **Example-prompt chips** — board-game-flavored questions; tapping one flows through the existing `onSend` → `handleSend` → `sendMessage` chain, which (via D-01) auto-creates a thread if none is active.
- **Composer-always-enabled posture** — the `ChatInput` stays enabled at all times (it is only `disabled={isStreaming}`, never gated on thread existence); the silent no-op is fixed by auto-create-on-send, NOT by disabling the input (D-01 rejected the "block input" alternative).
- **Visible first-message feedback** — the contract requires the optimistic user bubble + streaming assistant placeholder to render on the very first send (not a silent thread creation).

**Out of scope (deferred — do NOT design here):**
- Backend thread/message APIs — unchanged; already support `POST /api/threads {}` create + backend title-from-first-message (D-03).
- Model selection / picker / per-thread model — Phases 13/15.
- Key-state dot, settings, balance, demo banner — Phases 10 (shipped) / 14 / 15.
- Any new chat capability, new tool, or new message rendering — none in this phase.
- A "block input until + New Chat" affordance — explicitly rejected by D-01.

**Non-UI sibling work in this phase (mentioned for completeness, not specified by this contract):** the auto-create-on-send wiring — establishing a thread id *before* `sendMessage` runs so the `if (!threadId …) return` guard at `useChat.ts:55` passes and the optimistic bubble renders. Placement (wrap `sendMessage` inside `ChatPage.handleSend`, reusing the `handleNewThread` `POST /api/threads {}` pattern, vs inside `useChat`) is **planner/executor discretion** per CONTEXT.md, and must avoid races with the `isStreaming`/`threadId` guard and optimistic-message ordering. This contract specifies what the user **sees**, not where the create call lives.

## Spacing Scale

Declared values (multiples of 4 — mirrors Tailwind's default 4px base; these are the values this phase uses, not a new scale):

| Token | Value | Tailwind class | Usage in this phase |
|-------|-------|----------------|---------------------|
| xs | 4px | `p-1` / `gap-1` | Inner gap if a chip carries a leading lucide glyph (icon ↔ label) |
| sm | 8px | `gap-2` / `mb-2` / `px-2` | Vertical gap between headline and sub-line; chip internal horizontal padding pairs with `py-2` |
| md | 16px | `p-4` / `gap-4` / `mb-4` | Existing empty-state container padding (`p-4` already on the scroll area); gap between the sub-line and the chip cluster; gap between chips |
| lg | 24px | `gap-6` / `mb-6` | Optional larger breathing room between the headline block and the example-chip cluster on desktop |
| xl | 32px | `p-8` | Not used this phase |
| 2xl | 48px | `h-12` | Mobile top bar height (locked Phase 06.1) — empty-state sits in the column below it; not re-specified |
| 3xl | 64px | — | Not used this phase |

**Exceptions:**
- **Touch targets: minimum 44px square** (`min-h-11`, i.e. ≥44px) for every tappable example-prompt chip on mobile — chips are interactive `<button>`s, so each must clear 44px (`py-2.5`/`py-3` + adequate line-height gets there; verify on a ≤768px viewport). The composer Send button keeps its existing `px-4 py-2` footprint.
- **Empty-state centering:** the block keeps the existing `h-full flex items-center justify-center` centering (`ChatContainer.tsx:35`). The chip cluster is constrained `max-w-md` (≈ 28rem) on desktop so chips don't stretch full-bleed; full-width within the column on mobile.

## Typography

Inherits the 4-size / 2-weight scale locked by `10-UI-SPEC.md` / `08-UI-SPEC.md`. No new sizes or weights.

| Role | Size | Weight | Line Height | Tailwind | Usage in this phase |
|------|------|--------|-------------|----------|---------------------|
| Display | 24px | 700 (bold) | 1.2 | `text-2xl font-bold` | Empty-state headline ("Ask about any board game"). Single line. |
| Heading | 16px | 600 (semibold) | 1.2 | `text-base font-semibold` | Example-chip label text (the question itself reads as a tappable affordance, slightly emphasized) — OR chips may use Body weight (planner discretion); do NOT exceed semibold. |
| Body | 14px | 400 (regular) | 1.5 | `text-sm` | Empty-state sub-line / one-sentence explainer beneath the headline; multi-line copy uses 1.5. |
| Label | 12px | 400 (regular) | 1.4 | `text-xs` | Optional "Try one of these" / hint label above the chip cluster, if used. |

**Hard constraints (inherited):**
- Maximum **4** sizes total. No `text-lg` / `text-xl` / `text-3xl` in new code.
- Maximum **2** weights total: 400 (regular) and 600/700 (semibold/bold). **No `font-medium`.**
  - ⚠️ **Known inherited violation to NOT propagate:** the current `ChatInput` Send button uses `font-medium` (`ChatInput.tsx:40`). This contract does **not** require fixing that pre-existing class, but any NEW element this phase adds (chips, headline, sub-line) MUST use only 400 or 600/700 — do not introduce a second `font-medium`.
- Body line-height **1.5** required for the sub-line and any chip label that can wrap to two lines.
- The empty-state currently renders `text-gray-500` (`ChatContainer.tsx:35`). The headline should step UP in contrast (see Color) so it reads as a welcome, not muted placeholder text; the sub-line may remain muted.

## Color

Inherits the **dark-only** palette anchored on Tailwind `gray-*` / `blue-*` — exactly as locked by `10-UI-SPEC.md`. The 60/30/10 split below matches production.

| Role | Value | Tailwind | Usage in this phase |
|------|-------|----------|---------------------|
| Dominant (60%) | `#030712` | `bg-gray-950` | The chat surface background behind the empty-state (matches `ChatPage` `bg-gray-950`); the empty-state has **no card** — it floats centered on the page background. |
| Secondary (30%) | `#1f2937` | `bg-gray-800` | **Example-chip resting fill** — chips reuse the assistant-bubble surface (`bg-gray-800`), reading as a neutral suggestion, NOT a sent message. Hover: `bg-gray-700`. |
| Accent (10%) | `#2563eb` | `bg-blue-600` / `text-blue-400` | **Reserved — see list below.** NOT used as the chip resting fill. |
| Border (chip outline option) | `#374151` | `border-gray-700` | Optional 1px chip border (`border border-gray-700`) if a bordered/outlined chip is chosen over a filled `bg-gray-800` chip — planner discretion, pick ONE treatment and apply it to all chips. |
| Muted text | `#6b7280` / `#9ca3af` | `text-gray-500` / `text-gray-400` | The sub-line / explainer (`text-gray-400`); the existing empty-state default is `text-gray-500` — keep muted for secondary copy. |
| Headline text | `#e5e7eb` / `#f9fafb` | `text-gray-200` / `text-white` | The empty-state headline steps up from the muted gray so the welcome reads clearly. |

**Accent (`bg-blue-600` / `text-blue-400`) reserved for, in this phase, exactly:**
1. The existing composer **Send** button (`bg-blue-600 hover:bg-blue-700` — unchanged, `ChatInput.tsx`).
2. **Optional** chip hover/focus affordance: a `hover:border-blue-500` or `focus:ring-blue-500` accent on the example chips to signal interactivity (matches the textarea's existing `focus:border-blue-500`). This is the **only** new accent use permitted, and it is decorative-on-interaction, not a resting fill.

It is **NOT** used for: the chip resting fill (that is `bg-gray-800` / bordered gray — a chip is a suggestion, not a primary CTA), the headline, or the sub-line. **Do not render the example chips as solid `bg-blue-600`** — that would (a) read as already-sent user bubbles and (b) blow the 10% accent budget by turning 2–3 chips into the loudest thing on screen.

**Destructive (`bg-red-*`):** **none in this phase.** There are no destructive actions in the empty-state. (Thread deletion lives in `ThreadSidebar` and is out of scope for this contract.)

**No new color enters this phase.** No amber (reserved for `DemoPill`), no green, no purple, no custom hex.

**Contrast note:** `text-gray-200` on `bg-gray-950` ≈ 13:1, `text-gray-400` on `bg-gray-950` ≈ 5.9:1, chip label `text-gray-100`/`text-gray-200` on `bg-gray-800` ≈ 10:1 — all clear WCAG AA. Spot-check with devtools during implementation.

## Copywriting Contract

All copy is **specific, warm, board-game-flavored, and free of HTTP/technical jargon**, aligned with the project's voice in `10-UI-SPEC.md`. Exact wording is **Claude's Discretion** per CONTEXT.md — the strings below are the locked **shape and intent**; the planner/executor may refine wording within this tone, but must not drift to generic ("Start chatting") or technical copy.

| Element | Copy | Notes |
|---------|------|-------|
| Empty-state headline (Display) | **Ask about any board game** | Warm, on-domain, single line. Replaces the muted text-only `<p>`. Wording is discretionary within the board-game-KB voice. |
| Empty-state sub-line (Body, `text-gray-400`) | Search rules, compare mechanics, or get a recommendation. | One sentence. Names the three things the agent is good at (mirrors the app's core value). Keep ≤ ~12 words. |
| Example-prompt chip 1 | **Compare Catan vs Wingspan** | A comparison (exercises multi-game reasoning). |
| Example-prompt chip 2 | **How do you win Azul?** | A rules lookup (exercises retrieval). |
| Example-prompt chip 3 | **Recommend a 2-player game** | A recommendation (exercises cross-library reasoning). |
| Anon-variant reconciliation | *(see decision below)* | The existing block branches `isAnon` and mentions the attached D&D 5e sample. |
| Primary CTA (composer) | **Send** | Unchanged existing label (`ChatInput.tsx`). The empty-state itself has **no** separate CTA button — the chips ARE the calls-to-action and the composer is always available. |

**`isAnon` branch reconciliation (resolves the CONTEXT.md open point — D-02 "preserve or consciously reconcile"):**

The existing code shows two different strings: anon users see a sentence mentioning "the sample D&D 5e quick-reference that's already attached," authed users see the bland "Send a message to start the conversation." **Decision for this contract: UNIFY the visual structure (headline + sub-line + the same 2–3 example chips) for both states**, and reconcile the anon-specific content as follows:
- **Both** anon and authed users get the same headline, sub-line, and the same three board-game example chips above.
- The anon-only mention of the attached D&D 5e sample is **preserved as an optional extra hint line** (Label, `text-xs text-gray-500`) shown ONLY when `isAnon` — e.g. "A D&D 5e quick-reference is already attached — try asking about it." This keeps the useful anon onboarding cue without maintaining two divergent empty-state layouts.
- Rationale: D-02 calls for one welcoming empty-state; diverging the entire layout by auth state re-introduces the maintenance split the fix is meant to remove. Planner may collapse the anon hint into a fourth example chip instead of a hint line if that reads cleaner — discretion, but the D&D sample cue must survive for anon users.

**Empty-state IS the "no data" state.** There is no separate "no threads" vs "no messages" screen — per D-04, every null-thread state (cold start AND post-delete) renders this same empty-state, and the composer is always live. The empty-state shows whenever `messages.length === 0`.

**Error state:** No new error copy in this phase. The existing chat-error surfaces (`useChat.ts` in-thread error bubble "The assistant ran into a problem. Try again, or rephrase your question." + the 4s error toast "The assistant didn't respond. Tap the message to retry.") are **unchanged and out of scope** — they already cover the post-send failure path, which is identical whether or not the thread was just auto-created.

**Destructive actions in this phase:** **none.** The empty-state introduces no destructive action and therefore no confirmation dialog.

**Example-prompt send behavior (resolves CONTEXT.md discretion point):** Tapping a chip should **send immediately** (call `onSend(chipText)` directly), matching the ChatGPT/Claude pattern referenced in D-01 and giving the instant visible feedback D-02/`<specifics>` require — the optimistic user bubble + streaming assistant render at once, and the auto-create path (D-01) establishes the thread first. (Prefill-then-edit is the rejected alternative; if the planner has a strong reason to prefill instead, that is a documented discretion override, but immediate-send is this contract's default because it best satisfies the "first message just works, visibly" success condition.)

## Accessibility

- **Example-prompt chips:** native `<button type="button">` — Tab-focusable, Enter/Space activatable, default Tailwind-4 focus-visible ring (or the `focus:ring-blue-500` accent option). Each chip's accessible name IS its visible question text — no `aria-label` needed.
- **Touch targets:** each chip clears 44px on mobile (see Spacing exceptions).
- **Headline / sub-line:** static text. The headline should be a real heading element (`<h2>` or `<h1>` depending on the page's existing heading order) so screen-reader users get the landmark; the sub-line is a `<p>`.
- **Composer always available:** because the input is never disabled by thread state, keyboard users can always type and send — no focus trap, no dead control. Optional autofocus on empty-state (discretion) should not steal focus disruptively on every re-render.
- **Color is never the sole signal:** chips read as interactive via shape/border/hover, not color alone; the headline/sub-line contrast difference is reinforced by size and weight, not hue.
- **Contrast:** all combinations clear WCAG AA (see Color § contrast note).

## Open Questions for Planner / Developer

These do not block the contract; they are explicit handoffs (all flagged "Claude's Discretion" in CONTEXT.md):

1. **Auto-create-on-send placement:** wrap `sendMessage` inside `ChatPage.handleSend` (reusing the `handleNewThread` `POST /api/threads {}` pattern, then `setActiveThreadId` before `sendMessage`) vs inside `useChat`. Must establish the thread id BEFORE `sendMessage` so the `useChat.ts:55` guard passes and the optimistic bubble renders. Reuse `loadThreads()` post-send (already in `handleSend`) for the backend title (D-03) — no placeholder "New Chat" title on create.
2. **Chip treatment — filled vs bordered:** pick `bg-gray-800` filled OR `border border-gray-700` outlined, apply uniformly to all chips. Both are within the contract; do not mix.
3. **Chip component extraction:** inline in `ChatContainer` vs a small `EmptyState.tsx` / `ExamplePrompts.tsx`. Either is acceptable.
4. **Anon hint placement:** a fourth `text-xs` hint line vs folding the D&D-sample cue into a fourth example chip — both preserve the anon onboarding cue.
5. **Composer autofocus on empty-state:** optional UX nicety (CONTEXT.md discretion); not required by this contract.
6. **Send-immediately vs prefill:** this contract defaults to **send-immediately** on chip tap (best satisfies the visible-first-message success condition); prefill is a documented override if the planner has a strong reason.
