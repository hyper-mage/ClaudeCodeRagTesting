# Phase 999.1 Plan 02: Chat Empty-State Example-Prompt Chips Summary

**Replaced the bland text-only chat empty-state with a welcoming headline + sub-line + 3 tappable board-game example-prompt chips (`EXAMPLE_PROMPTS`) that send through the existing `onSend` chain, plus full component-test coverage.**

## Accomplishments
- Rebuilt the `ChatContainer.tsx` empty-state: `text-2xl font-bold` headline, `text-sm text-gray-400` sub-line, and a vertical cluster of 2-3 `bg-gray-800` chips (`min-h-11` ≥44px touch target), all inside the `{messages.length === 0 && (…)}` guard.
- Defined `EXAMPLE_PROMPTS` as a module-level `UPPER_SNAKE_CASE` constant (Compare Catan vs Wingspan / How do you win Azul? / Recommend a 2-player game) — each chip is a native `<button type="button" onClick={() => onSend(q)}>`, routing through the single existing send path.
- Unified the anon/authed layout (same headline + sub-line + chips for both); preserved the anon onboarding cue as a `text-xs text-gray-500` D&D-sample hint gated on `isAnon` only.
- Authored `ChatContainer.test.tsx` (5 tests) covering chip render by role+accessible-name, chip tap → single `onSend` with exact text, anon-hint reconciliation (hint present for anon, absent for authed, chips shared by both), and empty-state gating (hidden when `messages` is non-empty).

## Decisions Made
- **Inline chips, no component extraction** — kept the empty-state inline in `ChatContainer` rather than extracting `EmptyState.tsx`/`ExamplePrompts.tsx`; keeps `files_modified` limited to the two planned files (Claude's Discretion per plan/UI-SPEC).

- **`font-semibold`/body weight only** — no `font-medium` introduced; the known inherited Send-button violation was not propagated.
