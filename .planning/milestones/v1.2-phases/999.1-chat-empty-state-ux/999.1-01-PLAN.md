<objective>
Stand up the frontend's first component-test runner (Wave 0 Nyquist infrastructure) and make `useChat.sendMessage` closure-proof so a message sent against a null-thread state actually fires.

Purpose: This phase fixes a silent no-op (a message typed before any thread exists is dropped at the `if (!threadId …) return` guard). The fix has TWO halves that live in different files; this plan owns the hook half (`useChat.ts`) plus the test infrastructure both halves depend on. Without an explicit-threadId send, the downstream auto-create wiring in Plan 03 cannot defeat the React stale closure (RESEARCH § Pitfall 1).

Output:
- A working vitest + @testing-library/react + jsdom setup with an `npm run test` script and a shared `renderWithProviders` helper.
- `useChat.sendMessage` accepting an explicit `threadId` (`opts?.threadId ?? threadId`) so a freshly-created thread id can be passed in by the caller.
- A `loadMessages` guard so the thread-switch effect does not clobber the optimistic bubble mid-send (RESEARCH § Pitfall 2).
- Behavioral tests proving both.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser SPA → FastAPI `/api/threads/{id}/messages` | Authenticated POST; user-typed `content` crosses here. Unchanged endpoint; the only new client behavior is supplying the thread id from a value the server itself issued. |
| test runner → mocked I/O | Tests mock apiFetch/apiStream; no real network, no secrets in test fixtures. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-999.1-01 | Tampering (IDOR) | sendMessage explicit `threadId` param | mitigate | The explicit id is only ever the value returned by the server's `POST /api/threads` for the authenticated user (Plan 03 passes it); the messages endpoint re-checks `user_id` + RLS. No client-minted id is trusted. This plan adds the param but does not source it from user input. |
| T-999.1-02 | Information Disclosure | useChat.test.tsx fixtures / SSE mock | accept | Test fixtures use synthetic ids (`t-new`) and no real bearer tokens or keys; mocks never hit the network. Low risk, no secrets. |
| T-999.1-03 | Denial of Service (self) | loadMessages isStreaming guard | mitigate | The guard is non-destructive (returns early) and cannot starve a send; it only prevents a redundant fetch from racing the optimistic append. No new unbounded loop. |
</threat_model>
