# Phase 999.1: Chat empty-state UX - Context

**Gathered:** 2026-06-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the silent no-op when a user sends a chat message while no thread is active. Today `useChat.sendMessage` (`frontend/src/hooks/useChat.ts:55`) early-returns `if (!threadId || isStreaming) return`, so a message typed before any thread exists is dropped without feedback (the optimistic user bubble is never even added). `ChatPage` starts with `activeThreadId = null` and only sets it on `handleNewThread`/select. Caught during Phase 3 UAT.

**In scope:** Making the first message "just work" by auto-creating a thread on send for any null-thread state (cold start AND after deleting the active thread), plus a welcoming empty-state in the message pane. Frontend-only UX fix.

**Out of scope:** Backend thread/message APIs (unchanged — already support create + title-from-first-message), model selection, any new chat capability.

</domain>

<decisions>
## Implementation Decisions

### Empty-state behavior (core fork)
- **D-01:** **Auto-create a thread on first send.** When the user sends a message with no active thread, the app silently creates a thread (`POST /api/threads {}`), sets it active, then sends the message into it — zero clicks, message "just works" (matches ChatGPT/Claude). The composer stays enabled at all times; the "block input until + New Chat" alternative was rejected.

### Empty-state visuals + copy
- **D-02:** **Welcoming empty-state with 2–3 tappable example prompts.** The empty message pane shows a short headline + a few example board-game questions (e.g., "Compare Catan vs Wingspan", "How do you win Azul?", "Recommend a 2-player game"). Tapping one prefills/sends it, which flows through the same auto-create path (D-01). This extends the existing empty-state block at `ChatContainer.tsx:34-42` (currently a text-only `<p>`). Copy/tone should align with the project's UI-SPEC copywriting contract (see canonical refs). The existing `isAnon` vs authed copy distinction at that block must be preserved or consciously reconciled.

### First-thread title flow
- **D-03:** **Create the thread untitled; let the backend title it from the first message.** Reuse the existing `handleSend` → `loadThreads()` refresh (`ChatPage.tsx:61-65`) that already surfaces the backend's auto-generated title. Do NOT set a placeholder "New Chat" title on create (avoids label flicker).

### Deleted-to-empty parity
- **D-04:** **One code path for every null-thread state.** Auto-create-on-send applies whenever `activeThreadId === null` — both cold start (no threads) and after `handleDeleteThread` clears the active thread (`ChatPage.tsx:53-59`). No second silent dead-end. This is the natural outcome of fixing the send path rather than gating on `threads.length`.

### Claude's Discretion
- Exact placement of the auto-create logic (in `ChatPage.handleSend` wrapping `sendMessage`, vs inside `useChat`) — planner/researcher decides; must avoid races with the `sendMessage` `isStreaming`/`threadId` guard and the optimistic-message ordering.
- Exact example-prompt wording and headline copy (board-game flavored, UI-SPEC tone).
- Whether to keep the `isAnon`-specific empty-state copy or unify it.
- Composer autofocus / focus management on empty-state.
- Whether example-prompt tap should send immediately or prefill the input first.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ChatPage.handleNewThread` (`ChatPage.tsx:44-51`): exact `POST /api/threads {}` → prepend to `threads` → set active pattern the auto-create path should reuse.
- `ChatPage.handleSend` (`ChatPage.tsx:61-65`): already reloads threads post-send to pick up backend titles — D-03 needs no new title logic.
- Existing empty-state branch (`ChatContainer.tsx:34-42`): the insertion point for D-02; already branches on `isAnon`.

### Established Patterns
- Backend is stateless; threads/messages created via `apiFetch('/api/threads', …)`. Backend already auto-titles from the first message (relied on by D-03).
- `useChat(activeThreadId)` re-keys on thread id; `sendMessage` is guarded by `threadId`/`isStreaming`. The auto-create must establish a thread id BEFORE the send so the guard passes and the optimistic user bubble renders.

### Integration Points
- New code connects at the `onSend` → `handleSend` → `sendMessage` chain and the `ChatContainer` empty-state render. Example-prompt taps reuse `onSend`.

</code_context>
