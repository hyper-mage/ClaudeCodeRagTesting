<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OpenRouter → backend | The exchange response body carries the `sk-or-v1-…` key (untrusted-to-log content) |
| backend → frontend | `/status` + exchange responses cross to the browser — must never carry the key |
| Text-to-SQL tool (`authenticated`) → DB | Must remain unable to read `user_api_keys` (Phase 9 SEC-02) |
| frontend → backend (exchange POST) | `code` + `code_verifier` arrive in the POST body under the Supabase bearer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-03 | Information Disclosure | Exchanged key returned to the frontend (plaintext or ciphertext) | mitigate | Exchange returns `{"connected": True}` only; `/status` returns booleans + masked tail + connected_at; `encrypted_key` is never selected. Test `test_exchange_upserts_and_hides_key` asserts `sk-or-v1` absent from `resp.text`; `test_status_returns_masked_only` asserts no key value. |
| T-10-04 | Information Disclosure | OpenRouter error body / SDK error string echoed into HTTP error payload or log | mitigate | `httpx.HTTPStatusError` caught → generic `HTTPException(502, "Couldn't complete the OpenRouter connection.")`; never `resp.text`, never `exc_info=True` on the response. Test `test_exchange_403_generic_error` asserts no `sk-or` in the error body. (Pitfall 1) |
| T-10-05 | Spoofing / Tampering | Key bound to the wrong user | mitigate | Upsert `user_id` comes from `Depends(get_user_id)` (JWT `sub`), not from the request body or the OpenRouter response — binds to `auth.uid()` server-side. (Pitfall 2) |
| T-10-06 | Information Disclosure / Elevation | SQL-tool regaining read access to `user_api_keys` | mitigate | keys.py uses the service-role client only; sql_service.py `ALLOWED_SQL_TABLES` untouched; no GRANT added. `test_sql_keys_lockdown.py` stays green. (Pitfall 6) |
| T-10-07 | Tampering | `code`/`code_verifier` sent over querystring/URL instead of POST body | mitigate | `ExchangeRequest` is a POST JSON body model; the router accepts `code`/`code_verifier` only via the request body, never query params. |
</threat_model>
