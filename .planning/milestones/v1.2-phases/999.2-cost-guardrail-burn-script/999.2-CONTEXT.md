# Phase 999.2: Cost-Guardrail Burn Script - Context

**Gathered:** 2026-06-30
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver a programmatic burn script (in `backend/scripts/`) that deliberately spends real
money on the **owner/demo-funding OpenRouter account** until it trips that account's **$0.10
cost guardrail**, then records what actually happens (calls blocked? email sent?). The purpose
is to discharge **SEC-03** — proving owner-key cost exposure is bounded by both an app-level
kill switch and a provider-side guardrail — which is the **hard gate** on enabling
`demo_fallback_enabled` in prod (Phase 15 / DEMO-01, DEMO-02).

**In scope:** the burn script + its safety bounds; an empirical record of guardrail behavior
(block + email outcome); a demonstration that both kill switches work; a written SEC-03
finding (PASS or BLOCKER).

**Out of scope:** building/shipping the demo-fallback UI or flag-flip itself (Phase 15);
changing the demo's pinned free model or the resolution logic (Phase 11 owns that); any
automated/recurring cost monitoring beyond this one-time verification.

</domain>

<decisions>
## Implementation Decisions

### Burn target & environment
- **D-01:** Burn against the **prod owner/demo-funding OpenRouter account** — the account
  behind `settings.resolved_llm_api_key` in prod (`.env.prod` / Fly `LLM_API_KEY`). It is the
  account that will actually fund the demo fallback, so SEC-03 must prove *that* account's
  ceiling, not a proxy. **Safe to trip now:** `demo_fallback_enabled` is still `False`, so
  blocking the account temporarily has **zero user impact** (no one is on the owner key).
- **D-02:** Operational model behind the gate (user's intent): the demo fallback is
  **owner-toggled ON for controlled demos**; the default path for everyone else is **BYOK**
  ("connect your own OpenRouter account"). The $0.10 guardrail + kill switch exist precisely to
  **bound owner-key exposure during those owner-initiated demo windows** (incl. anyone using it
  during a demo without the owner's knowledge).

### Guardrail precondition
- **D-03:** The $0.10 cost guardrail is **already configured/live** on the prod owner account
  (confirmed by user). The phase verifies it trips + blocks; a defensive "verify the guardrail
  exists first" step is acceptable but treat configuration as present. (Prior intel: OpenRouter
  **Guardrails** min threshold is **$0.10**, **$5 min top-up**, and there is **NO email-alert
  toggle** — Guardrails replaced the old email-alert mechanism. See canonical refs.)

### SEC-03 done-bar (what "verified" means)
- **D-04:** SEC-03 is verified when **all three** hold: (1) the guardrail **BLOCKS** further
  owner-key calls after ~$0.10 cumulative spend; (2) the kill switch demonstrably **stops
  owner-key spend** (see D-05); (3) the **email-on-trip outcome is empirically recorded** —
  **"no email" is an ACCEPTED valid finding** (expected, per prior intel). Do **NOT** gate
  closure on a positive email confirmation — if guardrails genuinely don't email, that's a
  documented result, not a failure.

### Kill switch (defense in depth)
- **D-05:** The "working kill switch" SEC-03 demands = **BOTH proven independently**:
  (a) **App-level** — `demo_fallback_enabled=False` halts the owner-key spend code path: the
  `chat.py` demo branch (≈ lines 166, 196-201) never executes, so **no completion is minted on
  the owner key**. This is the instant, in-our-control off switch.
  (b) **Provider-level** — OpenRouter's **$0.10 guardrail** caps account spend as the backstop
  if the flag is ever bypassed or misconfigured (e.g., demo pointed at a paid model, or the
  free model unavailable).

### Spend ceiling & abort
- **D-06:** Drive spend just past **$0.10** using the **paid** `openai/gpt-4o-mini`
  (~$0.005/call). The demo path's normal **free** model ($0) cannot trip a dollar guardrail, so
  the test deliberately uses a paid model against the **same account**. **Bound overshoot** by
  minting **small serial-ish batches** and **re-checking cumulative spend between batches** (not
  one big parallel fan-out). **HARD ABORT at $0.25** (2.5× threshold): if no block by then,
  declare the guardrail **NOT working → SEC-03 BLOCKER** (do not enable demo in prod) and stop.

### Claude's Discretion
- **How the script mints paid calls** — either (i) a **standalone** OpenRouter chat-completions
  call using the owner key + `openai/gpt-4o-mini` (simplest; the guardrail is **account-level**,
  so it trips regardless of code path), or (ii) forcing the actual demo-fallback path with a paid
  model override (more faithful but heavier). Planner/researcher picks; (i) is sufficient for
  SEC-03 since the ceiling is account-scoped.
- **Spend-reading method** between batches — `GET /api/v1/key` usage delta vs the credits page.
- Exact batch size / minimal concurrency / inter-batch sleep, within the D-06 bounds.

</decisions>

## Existing Code Insights

### Reusable Assets
- `backend/scripts/_lib/` + `verify_langsmith_routing.py`: existing standalone-script bootstrap
  (load settings, build provider client) to copy for the burn script.
- `backend/config.py` `resolved_llm_api_key` + `llm_base_url`: the owner key + OpenRouter base
  URL the script authenticates with.
- `backend/services/llm_service.py`: canonical OpenRouter chat-completion call shape.

### Established Patterns
- Scripts live in `backend/scripts/`, run under the backend `venv`, read config from the active
  env file (`.env` dev / `.env.prod` prod — `ENV_FILE` override). **This script must target the
  PROD owner account (D-01)** → run with `ENV_FILE=.env.prod` (or an explicit owner-key arg),
  NOT the dev `.env`.
- Secret handling: the owner `sk-or-…` key must never be logged/printed (SEC-01 family). Reuse
  existing `scrub_secrets` discipline; the script prints spend + block status, never the key.

### Integration Points
- The script is **out-of-band** — it does not touch the FastAPI request path; it calls OpenRouter
  directly with the owner key. The only app-code link is conceptual: it validates the ceiling that
  bounds `chat.py`'s `demo_fallback_enabled` branch.
