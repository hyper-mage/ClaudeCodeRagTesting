# 06.1 CONTEXT — mobile-responsive-chat-layout

**Phase:** 06.1 (URGENT, inserted after Phase 6)
**Milestone:** v1.1 Portfolio Deployment
**Goal (from ROADMAP):** Chat + Documents pages usable on mobile viewports (≤768px). Current state: `IconSidebar w-14` (56px) + `ThreadSidebar w-64` (256px) = 312px of fixed left chrome on a 375px screen leaves ~63px usable for chat content. Discovered during 06-04 friend-testing.
**Date captured:** 2026-05-14

## <domain>

What this phase delivers: chat UI (and documents UI) collapses both desktop sidebars below `md:` (768px) into a slide-in hamburger drawer, restoring full viewport width to the active content (chat thread / documents list) on phone-sized screens.

What it does NOT deliver: new features, redesign, dark/light theme work, tablet-optimized layout beyond drawer toggle, mobile-specific gestures inside chat content (swipe-to-delete messages, etc.).

## <decisions>

### Sidebar collapse strategy
**Hamburger drawer.** Below `md:` (768px), both `IconSidebar` and `ThreadSidebar` hide. Hamburger button in a mobile-only top bar opens a slide-in drawer from the left edge (~280–320px wide) containing both navigation rails stacked.

Rationale: matches established mobile patterns (ChatGPT, Claude.ai, Slack). Drawer slide-in restores full viewport to chat content. Collapse-to-icons rejected (still cramps content below 400px); hybrid rejected (56px IconSidebar still useful screen real estate to surrender on phone).

### Breakpoint + scope
**Breakpoint:** `md:` = `768px` (Tailwind default).
**Scope:** `ChatPage` AND `DocumentsPage`. Both consume `AuthenticatedLayout` (`App.tsx`) which renders the persistent `IconSidebar`, so the responsive fix applies to the layout + both routes uniformly.

Rationale: `md:` covers phone + small-tablet; tablets ≥768px keep desktop two-rail layout. Including DocumentsPage in scope closes the milestone cleanly — leaving it mobile-broken would just become 06.2.

### IconSidebar render inside drawer
**Stacked top.** Inside the drawer:
- Top: horizontal row of `IconSidebar` nav items (Chat icon, Documents icon, Logout) — rendered as a row instead of a column on mobile
- Below: full-height `ThreadSidebar` (threads list) when on ChatPage, or `FolderTree`/equivalent when on DocumentsPage

Rationale: familiar pattern (top icons = global nav, body = current page's context list). Tap a nav icon → navigate + auto-close drawer.

### Drawer dismiss behavior (full kitchen sink)
All of the following enabled:
- **Backdrop click:** dimmed overlay behind drawer dismisses on click/tap
- **Swipe-left:** touch drag past threshold (e.g., 50px or 30% of drawer width) closes drawer. Hand-rolled with `pointerdown`/`pointermove`/`pointerup` — no `framer-motion` dependency
- **Escape key:** `useEffect` listening for `Escape` keydown while drawer open
- **Auto-close on select:** tapping a thread row, nav icon, or "+ New Chat" closes drawer AND performs the action
- **Close button (X):** top-right of drawer for explicit dismiss

Rationale: portfolio-polish target; users hit any of these conventions interchangeably. Hand-rolled gestures keep dependency footprint small (no framer-motion). CSS `transition-transform` handles slide animation; no JS-driven animation needed.

### Hamburger entry point
Mobile-only top bar appended to the top of `ChatContainer` / `DocumentsPage` body, visible only below `md:` via Tailwind `md:hidden`. Contains:
- Left: hamburger icon (lucide `Menu`) → opens drawer
- Center: page title ("Chat" / "Documents") OR current thread title (truncate)
- Right: optional `+ New Chat` shortcut (or omit; drawer has it)

Above `md:` the bar is hidden, desktop layout is unchanged.

### What stays the same
- Desktop layout (≥768px) is **not modified**. `IconSidebar` w-14 + `ThreadSidebar` w-64 + chat flex-1 unchanged. Phase is additive responsive overlay.
- `AuthenticatedLayout` in `App.tsx` remains `flex h-screen` but `IconSidebar` gets `hidden md:flex` to hide on mobile (its content is mirrored inside the drawer).
- `ThreadSidebar` gets `hidden md:flex` (or moved inside a conditional drawer container).
- Color tokens, spacing scale, dark theme — unchanged.

## <code_context>

### Reusable assets
- `lucide-react` already imported in IconSidebar — use `Menu` and `X` icons
- Tailwind 4 utility classes already in use: `hidden`, `md:flex`, `md:hidden`, `flex-1`, `w-64`, `w-14`, `bg-gray-900`, `bg-gray-950`, `border-r border-gray-800`
- React state pattern (`useState`) used throughout `ChatPage` / `DocumentsPage` — same pattern fits a `[isDrawerOpen, setIsDrawerOpen]` hoist in `AuthenticatedLayout` or per-page

### Integration points
- `App.tsx::AuthenticatedLayout` — hoist drawer state here so both ChatPage + DocumentsPage share one drawer (or per-page if simpler)
- `IconSidebar` + `ThreadSidebar` — extract their child content so they can be rendered both as desktop rails AND inside the mobile drawer without duplication
- Body scroll lock when drawer open (toggle `overflow-hidden` on `<body>` via `useEffect`) so backdrop doesn't scroll page underneath

### Risks / pitfalls
- `h-screen` + iOS Safari toolbar: `100vh` on mobile Safari includes hidden URL bar → cuts off content. Mitigate with `h-dvh` (Tailwind 4 supports `h-dvh` for dynamic viewport height) at minimum on mobile breakpoint.
- Swipe gesture inside chat (e.g., copy message) could conflict with drawer-close swipe. Scope drawer-close swipe to drawer surface only, not chat content area.
- `IconSidebar` currently `h-screen` — wrap may need restructure so it renders both as fixed rail (desktop) and as a non-screen-height row (mobile drawer top).
- React Router `<Link>` clicks inside drawer must trigger auto-close BEFORE navigation (so on back-navigation the drawer state is closed).

## <deferred>

- **Bottom-tab nav alternative for mobile** — could replace hamburger entirely (Chat / Documents tabs at bottom of viewport). Better one-handed UX. Not in scope; reconsider in v1.2 if user feedback prefers bottom-tabs.
- **Tablet-optimized layout (≥768px <1024px)** — currently tablets get desktop layout. Could squeeze sidebars (e.g., `lg:w-64 md:w-48`) for tablet polish. Defer to v1.2 polish phase.
- **Drawer swipe-to-open from edge** — current scope is swipe-to-close (drawer-only). Edge-swipe-open requires global edge gesture handler; defer.
- **Animation polish (spring physics)** — current scope is CSS `transition-transform`. `framer-motion` would enable spring animations. Defer unless v1.2 picker UI also wants animation lib.
- **Responsive fix for `LoginPage`** — login is already mostly OK on mobile (single column form). Verify during execution; if not OK, fold into this phase since it's trivial; otherwise note as deferred.

## <success_criteria>

1. Open `https://boardgame-rag-prod.pages.dev` on a 375px-wide viewport (Chrome DevTools mobile emulator) → chat input visible, message bubbles span full width, no horizontal scroll.
2. Hamburger icon (top-left of body) tappable; drawer slides in from left within ~200–300ms.
3. Drawer shows nav icons row at top (Chat, Documents, Logout) + thread list below.
4. Tap thread → drawer closes + thread activates.
5. Tap backdrop → drawer closes.
6. Swipe drawer left past threshold → drawer closes.
7. Press Escape (with keyboard attached) → drawer closes.
8. Body does not scroll when drawer is open.
9. ≥768px viewport: desktop layout unchanged; no hamburger button visible.
10. DocumentsPage on mobile: same hamburger pattern works, folder tree accessible via drawer.
11. No new npm dependencies introduced (no framer-motion).
12. iOS Safari 17+: drawer height respects dynamic viewport (no URL-bar gap).
