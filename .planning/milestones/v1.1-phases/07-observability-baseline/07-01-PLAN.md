<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Sentry SaaS | Error events POST to Sentry DSN over HTTPS. Untrusted user JWTs/email may be present in headers/breadcrumbs unless scrubbed. |
| CF Pages build env → Sentry SaaS | `SENTRY_AUTH_TOKEN` (project:releases + org:read scopes) uploads source maps + creates releases at build time. Token MUST NOT enter the runtime bundle. |
| dist directory → CF CDN | Source map files (`*.map`) generated by Vite at build time. If not deleted after upload, become publicly fetchable from the CDN. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | `event.request.headers.authorization` carrying a Supabase JWT (which embeds user UUID + email + role) shipped to Sentry | mitigate | `beforeSend` iterates request headers and replaces any lowercase `authorization` key with `'[redacted]'` before the event leaves the browser (RESEARCH Pitfall 3, Code Example §1) |
| T-07-02 | Information Disclosure | Fetch/xhr breadcrumb captures Authorization header in `breadcrumb.data.request_headers` | mitigate | `beforeBreadcrumb` scrubs request_headers.authorization for `fetch`/`xhr` categories (Pitfall 3) |
| T-07-03 | Information Disclosure | Console breadcrumb leaks `sb-<ref>-auth-token` localStorage key contents | mitigate | `beforeBreadcrumb` returns `null` to drop console breadcrumbs whose message matches `/sb-[^-]+-auth-token/` |
| T-07-04 | Information Disclosure | Source map files publicly served from `https://<app>.pages.dev/assets/*.js.map` enabling bundle reverse-engineering | mitigate | `sourcemaps.filesToDeleteAfterUpload: ['./dist/**/*.map']` deletes maps from dist after upload to Sentry (Pitfall 2). Plan 07-04 includes a deployed-bundle `curl -I` regression check. |
| T-07-05 | Information Disclosure | `SENTRY_AUTH_TOKEN` (build secret) leaks into the runtime bundle if accidentally placed in Runtime scope OR referenced via `import.meta.env` | mitigate | Plan reads token only via `process.env.SENTRY_AUTH_TOKEN` in `vite.config.ts` (Node-side); 07-04 sets it in CF Pages **Build** scope, never Runtime; SEC-07 leak-grep regression re-runs against deployed bundle in Plan 07-04 |
| T-07-06 | Spoofing | `Sentry.setUser({ email, id })` attaches user identity to events, contradicting CONTEXT D-1 anonymity rule | mitigate | `grep -rn "Sentry.setUser" frontend/src/` MUST return zero matches (enforced in acceptance criteria of Task 2 and verification block below) |
| T-07-07 | Denial of Service (self-inflicted) | `tracesSampleRate: 1.0` + session replay blows through Sentry free-tier 5k errors/month quota | mitigate | `tracesSampleRate: 0.1`, no replay integration (Pitfall 8 — values copied from RESEARCH §Code Example §1) |
</threat_model>
