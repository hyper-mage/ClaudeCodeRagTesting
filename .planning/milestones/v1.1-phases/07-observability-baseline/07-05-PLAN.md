<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 4: Simulated downtime drill (success criterion #3)</name>
  <what-built>
    Drill that forces Monitor 1 into Down state, confirms an email alert reaches the owner within one polling cycle, then reverts and confirms a recovery email also arrives.

    **Drill technique:** flip the Fly secret `SUPABASE_URL` to a bogus value so the supabase-py `select` head-only probe in `/api/health` raises and the handler returns 503 + `{"status":"degraded"}`. The keyword `"status":"ok"` is absent → UptimeRobot fires alert. Revert by setting `SUPABASE_URL` back to the real value. This exercises the failure path without taking the machine down (process stays up, cold-start unchanged), which gives the cleanest signal that the DB-reachability probe is doing real work.
  </what-built>
  <how-to-verify>
    1. Note current UTC time: `date -u +%Y-%m-%dT%H:%M:%SZ` — record as drill start.
    2. Capture current Fly secret value (do not echo it): `flyctl secrets list -a boardgame-rag-prod | grep SUPABASE_URL` (just confirms it exists; value is masked).
    3. Flip to bogus: `flyctl secrets set SUPABASE_URL=https://invalid.supabase.co -a boardgame-rag-prod` — this triggers a Fly deploy with the bad secret.
    4. Wait for Fly deploy to finish (`flyctl status -a boardgame-rag-prod` shows the new version healthy with the bad secret).
    5. Manually `Check Now` on Monitor 1 in UptimeRobot to skip the up-to-5-min poll wait. The check should return 503 (keyword missing) → monitor transitions to "Down".
    6. Confirm `<your-email>` inbox receives a "Monitor is DOWN" email from UptimeRobot within ≤10 min of the down transition.
    7. Revert: `flyctl secrets set SUPABASE_URL=<real-value-from-1Password> -a boardgame-rag-prod`. Wait for redeploy.
    8. Manually `Check Now` again — monitor returns to "Up".
    9. Confirm `<your-email>` inbox receives a "Monitor is UP" recovery email.
    10. Note drill end UTC time.

    Reply: drill start UTC, drill end UTC, down-alert email arrival timestamp, recovery-alert email arrival timestamp, and confirm both monitors are back to Up.

    **Safety:** keep the real `SUPABASE_URL` value handy (1Password) before flipping. If revert fails, copy from `.env` at repo root (un-prefixed `SUPABASE_URL` line per Phase 3 SUMMARY).
  </how-to-verify>
  <resume-signal>Type "approved" with the 4 timestamps + recovery confirmation, or list issues.</resume-signal>
