<objective>
Add a SMALL suite of REAL-DB integration smoke tests for the DB-shape backend endpoints
(`GET/PATCH/DELETE /api/threads/{id}`, `GET /api/folders/{id}/contents`, `GET /api/documents`)
so PostgREST-shape regressions — like the `get_thread` `maybe_single()` + `*, messages(*)`
resource-embedding APIError-204 bug that shipped toward prod this session — are caught by
`pytest` BEFORE deploy, not by reading a 500 in the server log.

The defining property: these tests do **NOT** patch/mock `get_supabase`. A mocked test passed
green while that exact code was runtime-broken; the only thing that surfaces this class of bug
is a real PostgREST round-trip. The canonical test creates a real thread + 2 messages, does
`GET /api/threads/{id}` (auth overridden, get_supabase REAL), and asserts 200 + both messages
present + ordered asc by created_at. On the broken pre-hotfix code that returns 500 (APIError
204); on the fixed code it returns 200 — that delta IS the guard.

Purpose: Turn a class of prod-bound 500s into a failing `pytest` before deploy.
Output: 4 new test files under `backend/tests/integration/` (skip-guarded, marker-tagged,
self-cleaning). No product code is modified.

> ⚠️ **THIS SUITE MUTATES THE DEV SUPABASE PROJECT.** It CREATEs and DELETEs throwaway rows
> (thread, messages, folder, document) in the `.env` dev project — the only real DB available
> (there is no separate test project per the dual-env setup). This is the accepted approach.
> Mitigations: (1) all rows are sentinel-prefixed `__inttest_260718__` (greppable/deletable),
> (2) fresh UUIDs per run, (3) teardown-after-yield always deletes them even if an assertion
> fails, (4) rows are owned by the never-login system user, (5) a skip-guard makes the whole
> suite a no-op when Supabase creds are absent (hermetic CI / other contributors).
</objective>

FK reality (from supabase/migrations) — the single most important constraint:
- threads / messages / folders / documents ALL have `user_id NOT NULL REFERENCES auth.users(id)`.
  A random UUID FAILS the FK. Use the SYSTEM USER `00000000-0000-0000-0000-000000000000`
  (migration 20240301000017_create_system_user.sql — a real, idempotent auth.users row that
  NEVER logs in). Deterministic, guaranteed present, no Supabase admin API needed.
- messages.thread_id REFERENCES threads(id) ON DELETE CASCADE  → deleting a thread deletes its messages.
- documents.folder_id REFERENCES folders(id) ON DELETE SET NULL → deleting a folder does NOT delete its docs.

Table columns to insert (NOT NULL unless noted):
- threads:   id(uuid), user_id(uuid), title(text, nullable), model(text,nullable), persona(text,nullable)
- messages:  id(uuid), thread_id(uuid), user_id(uuid), role(text CHECK in 'user'|'assistant'),
             content(text), created_at(timestamptz, has default — override for deterministic order)
- folders:   id(uuid), user_id(uuid), name(text), path(ltree — label chars [a-z0-9_] only),
             parent_id(uuid,nullable), visibility(text CHECK 'public'|'private', default 'private'),
             UNIQUE(user_id, path)
- documents: id(uuid), user_id(uuid), filename(text), storage_path(text, NOT NULL — dummy string ok),
             file_size(int), mime_type(text), status(text CHECK pending|processing|completed|failed),
             folder_id(uuid,nullable), visibility(text default 'private')
