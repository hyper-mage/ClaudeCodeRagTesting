# Quick Plan — Optimize UI Responsiveness & Backend Latency

## Complexity

⚠️ **Medium** — Nine behavior-preserving perf refactors. No new features, no new deps. Real regression risk lives in three places (SSE streaming under GZip, thread 404/no-clobber semantics under query-collapse, auth token freshness under session caching). Each is gated by an explicit flow-exercising verify, not a typecheck. Executable in one pass if the verifies are run honestly.

## Objective

Reduce per-request backend latency and eliminate redundant frontend work across the ChatPage mount fan-out and thread/folder navigation, WITHOUT changing any observable behavior. All nine fixes are scoped and file:line-located. Group into 5 atomic, independently-committable tasks with exclusive file ownership (no two tasks touch the same file → any task can be reverted alone).

**Hard constraint (user):** must not break existing functionality. Every documented invariant below is load-bearing (encodes a prior bug fix) and must survive unchanged.

## Invariants That MUST Survive (do not regress)

- **WR-01** (useChat): re-entrancy guard reads `isStreamingRef.current`, not the `isStreaming` state; `sendMessage` deps stay `[threadId, showToast]`.
- **CR-01 / skipNextLoad** (useChat): the `skipNextLoadRef` early-return path for a freshly auto-created thread must NOT abort the in-flight send and must NOT load `[]` over the optimistic user bubble.
- **abort-on-switch** (useChat): a genuine thread switch aborts the previous stream before rendering/loading the new thread.
- **exclude_unset no-clobber (T-17-05) + explicit-null clear (D-05/D-10)** (threads PATCH): a persona-only PATCH never writes `model`; a model-only PATCH never writes `persona`; an explicit `{model: null}` still writes `None`.
- **IDOR re-check (T-13-IDOR / T-17-04)** (threads GET/PATCH/DELETE): a non-owned / nonexistent thread returns 404; no cross-user row is ever read, written, or deleted.
- **DELETE 204** (threads): successful delete returns 204, no body.
- **service-role client is shared-safe by design** (database): it bypasses RLS server-side; no per-request state is mutated on the client object.
- **auth token freshness** (api.ts): never send an expired bearer; the logged-out (no-token) path sends no `Authorization` header.
- **refreshTree freshness** (useFolderTree): after upload/move/delete/rename the shown contents must be fresh, not a stale cache hit; public/private virtual-root logic unchanged.
- **auto-title** (ChatPage): the sidebar shows the backend-generated title after the FIRST message of an untitled thread.
- **live streaming bubble + tool cards** (MessageBubble): the last (assistant) bubble still updates per SSE delta; tool cards still update mid-turn.

## Task 3 — Frontend: cache the Supabase session in api.ts (fix 6)

<files>
frontend/src/lib/api.ts

<action>
`buildHeaders()` calls `await supabase.auth.getSession()` on EVERY `apiFetch`/`apiStream` — supabase-js v2 acquires a navigator LockManager lock per call (latency, worse with multiple tabs). Cache the session in module scope and read the token synchronously, with a correctness-preserving near-expiry fallback:
- Module-scoped `let cachedSession: Session | null = null` and `let initialized = false` (import the `Session` type from `@supabase/supabase-js`).
- Subscribe once at module load: `supabase.auth.onAuthStateChange((_event, session) => { cachedSession = session; initialized = true })`. This keeps the cache fresh on SIGNED_IN / SIGNED_OUT / TOKEN_REFRESHED / INITIAL_SESSION.
- In `buildHeaders`: if `!initialized` (subscription hasn't fired yet on first paint), do a one-time `const { data } = await supabase.auth.getSession(); cachedSession = data.session; initialized = true`. Else if `cachedSession` exists and is at/near expiry — `cachedSession.expires_at` (unix seconds); if `cachedSession.expires_at * 1000 - Date.now() < 60_000` — fall back to `await supabase.auth.getSession()` (which auto-refreshes) and overwrite `cachedSession`. Otherwise read the token synchronously from `cachedSession`.
- Read `const token = cachedSession?.access_token`; keep the existing `if (token) headers['Authorization'] = ...` and the no-token path unchanged. `buildHeaders` stays `async` (the near-expiry branch may await) — no call-site changes.
- MUST preserve: never send an expired token (near-expiry fallback covers it); the logged-out path (cachedSession null → no Authorization header).
</action>
